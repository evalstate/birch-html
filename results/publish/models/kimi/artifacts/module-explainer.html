Module explainer

Rendering checker pipeline

How Birch HTML artifacts are validated: from static contract checks through browser screenshots to geometry audits and structured reports.

Python 3 Chrome headless Pillow JSON + Markdown reports
Mental model

The checker is a deterministic gate, not a visual diff suite

Think of the pipeline as three layers that run in order, each layer stricter than the last:

  1. 1

    Static contract parsing

    Read the HTML file with html.parser. Verify doctype, viewport, Birch CSS presence, class counts, and component child contracts. No browser needed.

  2. 2

    Screenshot capture

    Launch Chrome headless, capture the rendered page at the requested viewport, then run palette and coverage metrics with Pillow. In --pair mode, compare original vs candidate screenshots.

  3. 3

    Browser geometry audit

    Inject a JavaScript audit into the page, measure overflow, alignment, squeezes, and wrapping pathologies that static parsing cannot see.

Key distinction

--artifact mode (used for eval outputs) captures evidence and flags egregious breakage. --pair mode compares a reference HTML with a Birch candidate and produces delta metrics. Do not mix the two flags.

Runtime path

From input pairs to reports, screenshots, and findings

Runtime flow in scripts/check_birch_renderings.py. Both modes converge on the same three validation layers before emitting reports.

Stage Entry function Key output Note
Parse + stats page_stats() PageStats Tag/class/data-attr counts, CSS vars, hex colors, Birch CSS validation.
Contract checks contract_findings() list[Finding] Doctype, viewport, Birch CSS, layout primitives, semantic components, child contracts, SVG presentation.
Screenshot capture() .png Chrome --screenshot; dynamic height for deep viewports via measure_document_height().
Screenshot metrics screenshot_metrics() dict Mean/RGB delta, diff PNG, contact sheet, palette closeness.
Geometry audit geometry_audit() dict Overflow, container spill, stat-card squeeze, metric alignment, timeline markers, code/text wrapping.
Report emit render_markdown() .json + .md Structured payload with summary, per-item findings, screenshot paths, and deltas.
File tour

Read these files in order before changing behavior

  1. 1

    styles/birch-system.css

    The canonical design system. The checker’s contract tests assume these tokens, primitives, and component child structures exist. If you change a component grid (e.g. .diff-row, .metric-row, .flow-step), update the matching parser in the checker.

  2. 2

    docs/birch-llm-style-guide.md

    The generation contract. Tells LLMs which classes to use, how to compose markup, and what the checker will enforce. If you tighten or relax a rule, document it here first so generators stay aligned.

  3. 3

    scripts/check_birch_renderings.py

    The checker itself. Start with main() to see CLI flags, then read check_artifact() and compare_pair() for the runtime path. Contract parsers live as HTMLParser subclasses near the top; geometry audit is a large injected JS string near the bottom.

  4. 4

    scripts/birch_mpl.py

    Matplotlib theme helper for chart artifacts. Defines Birch colors, rcParams, and SVG serialization. Used by chart evals to generate inline SVG that the checker later validates for fill="none" and stroke presence.

  5. 5

    evals/charts/build_chart_brief.py

    A concrete artifact generator that exercises the numeric-data recipe. It reads evals/charts/sample-data.csv, builds a Matplotlib SVG, and writes 24-birch-chart-brief.html. Run this to produce a real artifact for checker testing.

  6. 6

    evals/charts/run_eval.py

    The eval runner. Rebuilds the chart brief, then invokes the checker across four viewports (desktop, mobile, deep, mobile-deep). This is the canonical way to exercise the full pipeline end-to-end.

Extension points

Where to change behavior without breaking existing evals

Safe to extend

Add a new contract parser

Subclass HTMLParser, feed it in contract_findings(), and emit a Finding. Existing artifacts without the new class are unaffected.

Safe to extend

Add a geometry audit probe

Extend the injected JS in geometry_audit(), read the new array in geometry_findings(), and map it to a Finding. Existing pages without the targeted selectors pass silently.

Safe to extend

Add a new viewport preset

Define it in run_eval.py or pass --viewport name:WxH directly. The checker accepts any name:WxH string.

Safe to extend

Add screenshot metrics

Modify artifact_screenshot_metrics() or screenshot_metrics() to compute new Pillow-based signals. Keep the existing keys for backward compatibility.

Gotchas
  • Do not change BIRCH_CSS_SIGNATURES or BIRCH_CSS_MIN_BYTES without updating the style guide and all existing artifacts. These are the gate for “has Birch CSS”.
  • Geometry audit uses a temp file with a <base> tag. If you refactor geometry_audit(), ensure relative stylesheet links still resolve or the audit runs unstyled and produces false negatives.
  • Child-contract parsers are strict. .diff-row must have exactly span.ln + span.mark + span.code. .metric-row must be .caption + .meter + code. .flow-step must be .flow-num + one wrapper. Relaxing these will let broken markup pass.
  • Do not mix --artifact and --pair. The CLI raises SystemExit if both are provided.
  • Deep viewports measure document height first. If measure_document_height() fails, the capture falls back to the viewport height and may clip long pages.
  • Chrome must be discoverable on $PATH. The checker searches google-chrome, chromium, etc. If none is found, visual checks are skipped with a note.
Commands and outputs

Concrete paths you will use

Run the chart eval end-to-end 
uv run evals/charts/run_eval.py

Rebuilds the chart brief, then checks it at four viewports.

Check a single artifact 
uv run --with pillow python scripts/check_birch_renderings.py --artifact 24-birch-chart-brief.html --out reports/check.json --markdown reports/check.md

Use this when iterating on a new artifact before adding it to an eval.

Check a pair 
uv run --with pillow python scripts/check_birch_renderings.py --pair original.html:candidate.html

Produces delta metrics, diff PNGs, and a contact sheet.

Outputs
  • reports/birch-rendering-check-{name}.json
  • reports/birch-rendering-check-{name}.md
  • reports/birch-screenshots/*.png
