Module explainer

Rendering Checker Pipeline

How scripts/check_birch_renderings.py validates Birch HTML artifacts — from static HTML parsing through optional Chrome screenshots and browser geometry audits to structured findings reports. Written for new contributors who need to modify or extend the checker without breaking existing eval runs.

check_birch_renderings.py birch_mpl.py run_eval.py
Mental model

A three-layer deterministic quality gate

Picture the checker as a funnel with three concentric layers. The static layer runs pure Python HTML parsers — no browser required — to validate Birch component contracts, embedded CSS validity, CSS-variable usage, and forbidden patterns (gradients, patch markers, SVG fill=none violations). This runs on any machine.

The visual layer (Chrome required) captures headless screenshots at the requested viewport. Pillow then samples every fourth pixel and measures how many fall within tolerance of Birch's canonical 12-colour palette (ivory, slate, clay, oat, olive, rust, sky, and five grays). Palette-close fraction below 40 % is a warning; non-background fraction below 2.5 % is a fail (blank render); blackish fraction above 8 % is a warning (unstyled SVG or code blocks).

The geometry layer (Chrome required) injects roughly 300 lines of audit JavaScript into a temporary copy of the artifact, loads it through --dump-dom, and reads layout measurements from the data-birch-audit attribute it writes to document.body. It catches overflow, metric-bar misalignment, stat-card squeeze, pathological text wrapping, and code-block underfill.

Artifact mode

Pass --artifact path to check a single generated output with no reference comparison. Use this for all eval outputs. Repeatable: pass the flag multiple times for multiple artifacts.

Source: check_artifact()

Pair mode

Pass --pair original:candidate to compare a reference page against a Birch candidate. Adds structural delta findings (word coverage, section-count drift, Birch class delta). Use only when a meaningful reference exists.

Source: compare_pair()

Exit codes

Exits 1 if any fail findings exist. Pass --fail-on-warn to also exit 1 on warnings. The two modes are mutually exclusive — passing both flags raises SystemExit in main().

Source: main() — last block

Runtime path

From input HTML to reports

Dashed borders = requires Chrome and Pillow. Source: scripts/check_birch_renderings.py — main(), check_artifact(), compare_pair().

  1. 1

    Parse arguments, find Chrome

    main() collects --artifact or --pair flags and resolves all paths relative to the repository root (two levels above the script via ROOT = Path(__file__).resolve().parents[1]). It reads styles/birch-system.css to extract the full set of defined CSS variables — these become the allowed set for the unknown-var check. find_chrome() probes PATH for google-chrome, google-chrome-stable, chromium, and chromium-browser in that order.

  2. 2

    Collect page statistics

    page_stats() reads the full HTML, strips the Birch system CSS block (matched by the data-birch-system attribute), and feeds the remainder through StatsParser, which accumulates tag counts, CSS class counts, data-attribute counts, linked stylesheets, and text-word tokens. It separately scans the raw HTML for CSS variables used and defined. The Birch CSS block is validated: it must be at least 10 000 bytes and contain all five signatures — --color-ivory, --font-serif, .page, .section, .card.

  3. 3

    Run static contract checks

    contract_findings() receives the PageStats dataclass and the set of system-defined CSS variables. It runs eight named checks (doctype, viewport, Birch CSS, .page shell, layout primitive count, semantic component count, no unknown CSS vars, no patch marker lines). It then checks for gradient backgrounds and hard-coded colors outside the Birch palette, and runs eight dedicated HTML parser subclasses for component-level contracts. Each parser produces a list of failing examples appended as Finding(level="fail", …).

  4. 4

    Capture screenshot (Chrome)

    If Chrome was found, capture() runs --headless=new --screenshot using the artifact's real file:// URI so relative stylesheet links resolve. For deep or mobile-deep viewports, capture_height_for_viewport() calls measure_document_height() first: it injects a small load-event script into a temp file, measures document.scrollHeight via --dump-dom --virtual-time-budget=300, then caps the capture at min(viewport_max, measured + 80) to avoid blank space at the bottom of short pages.

  5. 5

    Measure pixel metrics (Pillow)

    artifact_screenshot_metrics() opens the PNG, down-samples to one-quarter resolution, and walks every pixel. It counts pixels close to the ivory background (delta ≤ 26), pixels close to any Birch palette colour (delta ≤ 26), and near-black pixels (R/G/B < 12). The resulting fractions drive three findings: blank_render fail if non-background fraction < 2.5 %, palette_drift warn if palette-close fraction < 40 %, black_pixel_fraction warn if blackish fraction > 8 %.

  6. 6

    Run geometry audit (injected JS)

    geometry_audit() prepends <base href="repo-root/"> inside <head> (so the shared stylesheet loads from the correct location) and appends the audit JavaScript before </head>. Chrome runs the modified page through --dump-dom --virtual-time-budget=800. The JS writes all measurements to data-birch-audit on document.body; the function regex-extracts and JSON-parses that attribute, then geometry_findings() converts each symptom into a Finding.

  7. 7

    Write reports and exit

    main() assembles the full payload dict, writes reports/birch-rendering-check.json (structured, with per-artifact stats, screenshot metrics, geometry data, and findings arrays) and reports/birch-rendering-check.md (Markdown table via render_markdown()). It prints both paths to stdout, then exits 1 if failures (or warnings with --fail-on-warn) exist.

File tour

Read these in order

  1. 1

    scripts/check_birch_renderings.py

    The main entry point. Start with main() to see argument parsing and mode selection. Then read check_artifact() (artifact mode) and compare_pair() (pair mode) to understand which sub-functions each calls. Work down through page_stats(), contract_findings(), and geometry_audit(). The eight HTML parser subclasses (StatsParser, MetricRowParser, DiffRowParser, DiffPolarityParser, FlowStepParser, GridListParser, ChecklistParser, TimelineParser) are the canonical definitions of every component contract the checker enforces. Finish with render_markdown() and geometry_findings().

  2. 2

    scripts/birch_mpl.py

    The Matplotlib SVG helper. Read COLORS and PALETTE first — these are the Birch hex values used for both chart colours and the pixel-distance palette check. Read rc_params() to understand the full chart theme (font, axis colours, grid, legend, transparent background). Then read birch_theme() (context manager), polish_axes() (removes top/right spines, adds grid), and svg_string(), which strips Matplotlib metadata and returns a bare SVG element ready for inlining. Callers patch in class="chart-svg" and role="img" via re.sub() after the call.

  3. 3

    evals/charts/build_chart_brief.py

    A complete artifact builder and canonical usage example. Shows the full loop: load CSV data from evals/charts/sample-data.csv, call birch_theme() as a context manager, call svg_string(fig) then patch the SVG tag with re.sub(), then f-string-compose the full HTML artifact to 24-birch-chart-brief.html in the repository root. Study metric_rows() and table_rows() for exact .metric-row and .numeric-table contract examples. The builder links styles/birch-system.css directly — the checker handles both linked and embedded forms.

  4. 4

    evals/charts/run_eval.py

    The eval orchestrator. Calls build_chart_brief.py first via uv run, then runs the checker four times with different --viewport flags: desktop:1365x900, mobile:390x900, deep:1365x2400, and mobile-deep:390x3000. Each run receives distinct --out and --markdown paths so reports do not overwrite each other. To add a new eval artifact: create a builder script, add viewport entries to the VIEWPORTS dict, and add the builder call at the top of main() before the checker loop.

Components

HTML parsers and what they enforce

StatsParser

Used by: page_stats()

  • Counts every HTML tag, CSS class, and data-* attribute.
  • Extracts the page title, text words, inline style values, and linked stylesheets.
  • Skips content inside <style> and <script> using a skip_depth counter to avoid false class/text matches.

MetricRowParser

Used by: contract_findings()

  • Validates that every .metric-row div has exactly three direct children: .caption, .meter, code.
  • A fourth child creates an extra implicit grid column, misaligning the value against its meter bar.

DiffRowParser + DiffPolarityParser

Used by: contract_findings()

  • DiffRowParser: checks each .diff-row has exactly span.ln + span.mark + span.code. Loose text in the row lands in the narrow line-number column and wraps to one character per line.
  • DiffPolarityParser: rows with a + mark must carry .add or data-kind=add so Birch applies the green colour token.

FlowStepParser

Used by: contract_findings()

  • Checks each .flow-step has exactly two direct children: .flow-num first, then one content wrapper.
  • Loose title or detail elements placed directly in the step (without a wrapper) can overflow into the narrow number column.

GridListParser + ChecklistParser

Used by: contract_findings()

  • GridListParser: items in .insight-list and .takeaway-list must wrap their content in exactly one direct child — these lists use CSS grid and extra direct children become accidental grid cells.
  • ChecklistParser: flags checklist items with multiple direct inline elements when the list uses flex layout.

TimelineParser

Used by: contract_findings()

  • Each .timeline-item must have both a .timeline-time child and a .timeline-body child, unless the parent timeline uses a custom scoped class.

svg_visual_findings()

Used by: contract_findings()

  • Scans all <polyline> and <path> elements with chart-line classes (chart-line, line-pass, trend-line).
  • Fails if the element lacks fill="none": the browser default fills the path solid black, producing a large wedge instead of a line chart.
  • Fails if no explicit stroke colour is set.

geometry_audit() — JS probes

Used by: check_artifact(), compare_pair()

  • Measures scrollWidth − clientWidth and getBoundingClientRect().right − viewportWidth for text, code, chips, badges, and stat-values.
  • Computes meter x-position spread within each .metric-list and flags if spread exceeds 2 px.
  • Detects stat-card squeeze (card width < 150 px or value clips right edge).
  • Checks .section-rail: aside rendered to the left of main content triggers a warning.
  • Also checks timeline marker geometry, code-block underfill, and pathological text wrapping.
Gotchas & extension points

Where to be careful and where to make changes

Common gotchas

Substitute the Birch CSS before checking

If an artifact still carries the unsubstituted Birch CSS placeholder in its data-birch-system style block, the embedded CSS will be far below the 10 000-byte minimum and the uses_birch_system_css check will fail. Run uv run skill/scripts/finish_birch_html.py <output.html> before invoking the checker — or link styles/birch-system.css via a <link> tag instead.

--artifact and --pair are mutually exclusive

Passing both flags in the same invocation raises SystemExit immediately in main(). All generated eval scripts use --artifact; --pair is only for manual comparison runs against a reference page.

Gradient backgrounds are a hard fail

linear-gradient, radial-gradient, and conic-gradient anywhere in page-local CSS produce a no_birch_gradients fail finding. Use flat Birch token surface colours instead.

SVG polylines require fill=none

Browsers default SVG fill to black. Any <polyline> or chart <path> without an explicit fill="none" attribute (or CSS equivalent) fails svg_visual_findings(). The result in a screenshot is a large black wedge rather than a line.

Geometry audit adds a base element to a temp copy

geometry_audit() writes a modified copy of the artifact to a tempfile.TemporaryDirectory and adds a <base href="repo-root/"> element so relative stylesheet links resolve. Artifacts that inline Birch CSS avoid this dependency entirely.

Deep viewport height uses a second Chrome pass

Viewport specs containing deep (e.g. deep:1365x2400) trigger measure_document_height(), which injects a tiny measurement script into a temp file and retrieves the natural page scroll height. The final capture height is clamped to min(viewport_max, measured + 80). Changing a viewport spec string changes which path executes.

Extension points

  • New component contract: write a new HTMLParser subclass modelled on MetricRowParser. Instantiate and feed it in contract_findings(). Append a Finding if the bad-items list is non-empty.
  • New geometry symptom: add a querySelectorAll block to audit_js in geometry_audit(), include the key in the JSON written to data-birch-audit, and add a handler in geometry_findings().
  • Track a new Birch class: add it to BIRCH_CLASSES, LAYOUT_CLASSES, or SEMANTIC_CLASSES at the top of the file. Change SEMANTIC_CLASS_MIN to adjust the warn threshold.
  • New palette colour: add it to PALETTE in check_birch_renderings.py (RGB tuple) and to COLORS in birch_mpl.py (hex string). Keep both in sync.
  • New eval artifact: create a builder script following evals/charts/build_chart_brief.py. In run_eval.py, add the builder call to main() and add entries to VIEWPORTS.
  • Redirect report paths: pass --out and --markdown. Both default to reports/ in the repository root. Use distinct paths when running multiple viewports.

Key thresholds

  • BIRCH_CSS_MIN_BYTES = 10_000 — minimum embedded CSS size
  • SEMANTIC_CLASS_MIN = 6 — minimum semantic class count (warn if below)
  • 0.40 — palette-close warn threshold in artifact mode (0.55 in pair mode)
  • 0.025 — non-background fraction fail threshold (blank render)
  • 0.08 — blackish fraction warn threshold
  • 2500 bytes — local CSS size warn threshold
Commands & outputs

Running the checker

Check a single artifact

uv run --with pillow python scripts/check_birch_renderings.py --artifact eval-runs/my-run/my-artifact.html

Check at mobile viewport

uv run --with pillow python scripts/check_birch_renderings.py --artifact eval-runs/my-run/my-artifact.html --viewport mobile:390x900

Compare reference vs. candidate

uv run --with pillow python scripts/check_birch_renderings.py --pair 05-design-system.html:05-design-system-birch.html

Run full chart eval

uv run python evals/charts/run_eval.py

Rebuild chart artifact, then check

uv run --with matplotlib python evals/charts/build_chart_brief.py && uv run --with pillow python scripts/check_birch_renderings.py --artifact 24-birch-chart-brief.html

Fail on warnings too

uv run --with pillow python scripts/check_birch_renderings.py --artifact eval-runs/my-run/my-artifact.html --fail-on-warn

Report outputs

File Description
reports/birch-rendering-check.json Full structured payload: mode, per-artifact stats, screenshot metrics, geometry audit data, and the findings array with level/name/evidence.
reports/birch-rendering-check.md Human-readable Markdown table of findings. Printed to stdout on completion.
reports/birch-screenshots/<stem>-<viewport>.png PNG screenshot per artifact per viewport. Only written when Chrome is available.
reports/birch-screenshots/<ref>__vs__<cand>-diff.png Amplified pixel-difference image (pair mode only). Brightness ×4 applied via Pillow.
reports/birch-screenshots/<ref>__vs__<cand>-contact.png Contact sheet: source, candidate, and amplified diff side-by-side (pair mode only).

Viewport spec format

The --viewport flag takes the form name:widthxheight. The name is used only as a label in report and screenshot filenames. The word deep anywhere in the name triggers the second-pass height measurement. Examples from evals/charts/run_eval.py: desktop:1365x900, mobile:390x900, deep:1365x2400, mobile-deep:390x3000.
