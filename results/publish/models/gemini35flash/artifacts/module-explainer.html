System Architecture

Mental Model & Visual Gates

The rendering checker acts as an automated visual gatekeeper for generated HTML documents. Rather than relying on computationally heavy visual-language models (VLMs) for basic sanity testing, the checker adopts a deterministic-first approach, dividing document validation into three increasingly strict execution gates:

1. Static HTML Contract Gate

Runs fast syntactic evaluations. Subclasses of HTMLParser extract element structures, verify nested layout rules, check CSS variables, and flag illegal style patterns (such as background gradients) before launching a browser.

2. Dynamic Browser Geometry Gate

Launches Headless Chrome to render the artifact. Injects custom diagnostic JavaScript (audit_js) to query element bounding rectangles on load, diagnosing overflow, squeezing, or vertical misalignments.

3. Visual Regression Gate

Performs screenshot capture. Uses Pillow to sample the visual color palette (verifying conformity with canonical Birch ivory/clay tones) and calculates Mean and RMS RGB distance metrics against original reference files.

Operating Modes

The suite supports two main execution pathways: Candidate-only Mode (using --artifact <path>) which renders and verifies a newly generated artifact in isolation, and Pair Comparison Mode (using --pair <ref>:<cand>) which computes visual regression deltas and contact sheets against a reference baseline.

Use these commands in your local shell to execute verification tests manually:

The Runtime Path in Detail

Follow the pipeline's algorithmic steps as it compiles findings:

  1. 1

    Option Parsing & System Initialization

    Resolves physical file directories, matches CLI viewport rules (e.g., desktop:1365x900), and loads the canonical CSS tokens directly from styles/birch-system.css to compile standard CSS variables.

  2. 2

    Static Syntactic Scans

    Reads the HTML in-memory. Subclasses of HTMLParser assess formatting structures. Specifically: StatsParser (toggles tag counters), DiffRowParser (checks line-num/mark/code cells), MetricRowParser (checks progress-bar columns), and FlowStepParser (ensures correct container wrapping). Also checks for inline gradient definitions and accidental '+' characters.

  3. 3

    Headless Chrome Boot & Height Adjustment

    Launches Chrome in headless mode. For viewport profiles containing 'deep' specifications, capture_height_for_viewport() registers a load listener and queries document.documentElement.scrollHeight dynamically to prevent cut-off captures.

  4. 4

    In-Browser Geometry Audit

    Injects audit_js into the page. The script utilizes browser bounding boxes to diagnose defects invisible to static files: document-level scrolling, squeezed stat boxes, misaligned sidebar containers, and mismatched numeric table column alignments.

  5. 5

    Pillow Screenshot & Palette Analysis

    Saves visual snaps. In pair mode, computes Mean and RMS RGB distances between reference and candidates, generating 4x-amplified difference sheets. In both modes, palette_metrics() resizes images to calculate canonical color density, flagging blank pages.

  6. 6

    Consolidation & Exit Signalling

    Compiles all anomalies into a detailed JSON payload (reports/birch-rendering-check.json) and a human-readable Markdown digest (reports/birch-rendering-check.md). Exits with exit code 1 on severe failures.

Onboarding

File Tour for New Contributors

To safely modify the rendering checker without breaking the existing test suites or failing candidate evaluations, follow this structured file tour in order:

Step 1
styles/birch-system.css (The Visual Grammar)

Read this first to master the design system's variables (e.g. --bg, --surface, --accent), spacing sizes (--space-1 to --space-9), and primitives (grids, splits, cards). It forms the layout dictionary the checker validates candidates against.

Step 2
docs/birch-llm-style-guide.md (The LLM Contract)

Understand the style contracts, nested grid rules, and design guidelines that LLM-generated artifacts are required to fulfill. It documents the exact structural rules that the checker's parser subclasses enforce.

Step 3
scripts/check_birch_renderings.py (The Execution Engine)

Study the parser's logic. Examine the HTMLParser subclasses (e.g., MetricRowParser, DiffRowParser) and explore the raw Javascript inside the geometry_audit() function to understand how elements are queried for layout errors inside the headless browser.

Step 4
scripts/birch_mpl.py (Visualization Helpers)

Inspect how Matplotlib's global rcParams are temporarily overridden (via the birch_theme() context manager) to match Birch color palette variables. Review the axes cleanup (polish_axes()) and SVG serialization helpers.

Step 5
evals/charts/build_chart_brief.py (Reference Artifact Creator)

Analyze this script to see an end-to-end implementation of Birch HTML generation. It reads raw sample CSV rows, creates visual charts with birch_mpl.py, formats metric rows, and compiles a clean, right-aligned numeric table.

Step 6
evals/charts/run_eval.py (Evaluation Test Runner)

Conclude your file tour by reviewing the automated evaluation script. It generates the chart brief and passes it to the rendering checker at desktop, mobile, deep, and mobile-deep viewports to validate cross-platform responsiveness.

Modification Guide

Gotchas, Modifications, and Extension Points

Critical Pipeline Gotchas

SQUEEZE

1. Direct Children in Flex/Grid Containers

Layout classes like .insight-list li, .takeaway-list li, .flow-step, and .diff-row use explicit grid layouts. Putting loose direct text or un-wrapped elements under these elements converts them into extra, anonymous grid columns. This squishes code text, forcing single characters to wrap in vertical trails. Fix: Always wrap multi-part text and details inside a single block container such as a <div> or <span>.

BLACK WEDGE

2. Default Fill in Custom SVG Path Elements

In SVG, <polyline> and <path> elements default to a solid black fill unless configured. Under browsers, this turns line charts into big solid black wedges. The checker statically catches this in svg_chart_mark_presentation. Fix: Always set an explicit fill="none" and configure a stroke color on custom line marks.

BASE PATH

3. Headless Chrome Relative Resource Failures

Headless Chrome runs the browser audit on a copy of the HTML inside a temp directory. This causes relative stylesheet links (e.g. styles/birch-system.css) to break, causing a complete lack of styles and throwing false-positive overflow failures. Fix: The checker dynamically detects the workspace root and injects an absolute <base href="..."> tag to preserve stylesheet resolution during the audit.

Pipeline Extension Points

MODIFY

Adding a New Static HTML Contract Check

Create a new parser class extending html.parser.HTMLParser inside scripts/check_birch_renderings.py (similar to MetricRowParser). Instantiate and run it inside the contract_findings() function. If a violation is found, append a Finding("fail", "your_check_name", "evidence detail") object to the return list.

MODIFY

Extending In-Browser Geometry Audits

To check for new layout issues (such as text overlapping page margins, icon dimensions, or un-clickable buttons), locate the audit_js string template inside scripts/check_birch_renderings.py. Add a DOM query, compute bounding rectangles via getBoundingClientRect(), flag any alignment defects, and append them to the final serialized JSON inside the data-birch-audit body attribute.

MODIFY

Updating the System Palette Tolerances

To expand the canonical color spectrum or adapt to a design token change, adjust the global PALETTE dictionary inside scripts/check_birch_renderings.py (which defines acceptable colors checked by Pillow's palette compliance functions) and coordinate those modifications inside the COLORS and PALETTE configurations in scripts/birch_mpl.py.
