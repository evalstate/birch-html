Blocking & High-Risk Findings

HTMLParser Stack Desynchronization on Void Elements

BLOCKING

File: scripts/check_candle_renderings.py | Class: TimelineParser | Line: ~530-550

Impact: In HTML5, void/self-closing elements (such as <img>, <br>, <hr>, or <input>) do not have closing tags. When the parser encounters them in handle_starttag, they are pushed onto self.element_stack. However, because no closing tag exists, handle_endtag is never called for them. This causes the next legitimate closing tag (like </div>) to pop the self-closing element's classes instead. This catastrophic desynchronization breaks downstream layout contracts and falsely flags valid markup as failing.

Evidence: Insecure element stack popping in check_candle_renderings.py
538- if "timeline" in classes:
539- self.timeline_stack.append(classes)
540- self.element_stack.append((tag.lower(), classes))
541+ # CRITICAL BUG: Pushes all tags including void tags (img, br, hr) without closing counterparts

Recommended Fix: Exclude known HTML5 void elements from being pushed to the stack in handle_starttag, or verify the tag name inside handle_endtag before popping:

VOID_ELEMENTS = {"img", "br", "hr", "input", "meta", "link"} if tag.lower() not in VOID_ELEMENTS: self.element_stack.append((tag.lower(), classes))

BLOCKING

Medium Stability & Logic Findings

Lexicographical Max Sorting on Numeric Weeks

WARNING

File: evals/charts/build_chart_brief.py | Function: latest_rows | Line: ~81-83

Impact: Converting week values to raw strings and using Python's max() results in a lexicographical sort. If week names cross standard boundaries (e.g. "Week 9" vs "Week 10", or raw numbers "9" vs "10"), the evaluation evaluates "Week 9" > "Week 10" (since '9' is larger than '1'). This causes the pipeline to lock to an outdated week or empty data sets, introducing severe data drift and silent reporting failures.

Evidence: Lexicographical max week query in build_chart_brief.py
82-def latest_rows(rows: list[dict[str, object]]) -> list[dict[str, object]]:
83- latest_week = max(str(row["week"]) for row in rows)

Recommended Fix: Implement a numeric parsing strategy or sort using parsed integer keys:

# Extract week number digits and max by integer value def get_week_num(row): match = re.search(r'\d+', str(row["week"])) return int(match.group()) if match else 0 latest_week = max(rows, key=get_week_num)["week"]

HTMLParser State Collision on Nested List Items

WARNING

File: scripts/check_candle_renderings.py | Class: GridListParser | Line: ~150-155

Impact: If an insight-list or takeaway-list contains a nested sub-list (e.g., a <ul> inside a parent <li>), the end tag of the nested list item (</li>) triggers the parser's handle_endtag for "li". Because the parser simply calls self.li_stack.pop() without validating depth, it prematurely pops the parent list item's state, corrupting state tracking and yielding unreliable contract checks.

Evidence: Unchecked list element popping in check_candle_renderings.py
152- elif tag == "li" and self.li_stack:
153- item = self.li_stack.pop() # <-- Prematurely pops outer parent on nested li close

Recommended Fix: Track list depth carefully and only pop when the closing tag matches the current depth, or transition to standard parsed element structures.

Unchecked Division by Zero on Empty Datasets

WARNING

File: evals/charts/build_chart_brief.py | Function: main | Line: ~115-120

Impact: The calculations for average static pass rates, average vision scores, and mean RGB deltas divide directly by len(latest). If the input CSV is empty, or if the week filter yields no matches, latest is an empty list, and the script immediately terminates with a ZeroDivisionError, crashing the entire build evaluation flow.

Evidence: Unprotected division by length in build_chart_brief.py
119- avg_pass = sum(float(row["static_pass_rate"]) for row in latest) / len(latest)

Recommended Fix: Add a simple presence guard:

if not latest: print("Warning: No matching week data found to generate chart brief.") return

Low-Risk Layout & Robustness Warnings

Split Rail Heading Layout Alignment Discrepancy

LOW

File: evals/charts/build_chart_brief.py | Hunk: Split section | Line: ~185

Impact: In the Trend section, the left side uses an <h2> element, and the right-hand reference panel uses an <h3> element inside .split. Due to differing margin and line-height values in the stylesheet, the tops of these headings do not align, triggering a split_rail_heading_alignment warning in the newly injected browser geometry audit.

Recommended Fix: Swap the right side heading to a matching <h2> and apply styling or wrappers to ensure consistent vertical baseline alignment.

KeyError Vulnerability on Missing Week Data

LOW

File: evals/charts/build_chart_brief.py | Function: build_svg | Line: ~60

Impact: The line chart generation script assumes every artifact is represented in every week of the dataset. If an artifact has missing data points for any given week, by_artifact[artifact][week] throws a KeyError, terminating the rendering execution pipeline.

Recommended Fix: Use .get() with a fallback or fill missing weeks with neutral values (e.g. None or 0.0) to prevent plot generator failure.
