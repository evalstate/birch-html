Implementation plan

Clean the benchmark inputs first, then lock rubrics, packaging, and the final run

Grounded in evals/implementation-plan/migration-plan.md: the safest sequence is Phase 1 → Phase 2 → Phase 3 → Phase 4, with verification built into the handoff between each phase. The maintainer’s first move should be to clean the benchmark inputs so later prompt, rubric, and packaging checks are stable.

Recommended first step

Start with Phase 1: clean benchmark inputs

Why this comes first

The source already places input cleanup ahead of rubric normalization, skill packaging, and the final benchmark. That order matters because later phases depend on stable eval paths, current package names, and a clear decision about what each prompt mode includes.

  • It removes stale or archived sources from active eval paths before the benchmark is used to judge the skill.
  • It forces the with-shell vs no-shell split to be decided up front instead of surfacing as a hidden conflict later.
  • It makes the remaining phases reproducible because the source set stops changing underneath them.

What the maintainer should do next

Update the benchmark source surface before touching the final benchmark run itself.

  • Keep each eval’s prompt, rubric, and source material together under evals/.
  • Remove historical or archived data sources from active eval paths.
  • Make the prompt mode decision explicit: with-shell path references only, or no-shell preview content where needed.

Confirmed facts, assumptions, and open questions

Confirmed facts

  • The current package is named birch-html, and the skill entrypoint is skill/SKILL.md.
  • The preferred workflow is: read sources, copy skill/resources/template.html, replace body content, run uv run skill/scripts/finish_birch_html.py <output.html>, then validate with skill/scripts/check_birch_renderings.py when available.
  • The artifact shell must remain <main class="page stack" data-gap="lg">...</main>, and the final HTML must embed Birch CSS in style[data-birch-system].
  • The benchmark set contains five evals: numeric-data, code-review, module-explainer, implementation-plan, and benchmark-comparison.
  • Source data lives under each eval directory, such as evals/code-review/source.diff and evals/numeric-data/source.csv.
  • The final benchmark should test the skill runner, not the older repo-mode migration prompt path.
  • Already-decided design constraints: keep .auto-grid as the card-grid primitive, prefer skill-based assembly now, keep recipes structural, and avoid Birchline-only classes.

Assumptions

  • The final benchmark is intended to evaluate the birch-html skill workflow.
  • Shell-enabled runs are allowed to inspect local repository files.
  • No-shell runs are optional, but they should remain runnable with source previews.
  • Semantic output scoring can wait until the generation and rendering pipeline is stable.

Open questions

  • Should the no-shell prompt set be part of the published benchmark or only a fallback/debug mode?
  • Which models should be included in the final published run?
  • Should semantic scoring be manual, model-judged, or hybrid?
Milestones

Phases, purpose, changes, inputs, and exit criteria

  1. 1

    Clean benchmark inputs

    Purpose: make the benchmark easy to understand and reproducible before anything else changes. Changes: keep each eval’s prompt, rubric, and source material together under evals/; remove historical or archived data sources from active eval paths; ensure prompts do not mention stale package names or deleted files; decide which prompts expose source previews and which require shell reads. Owner / inputs (inferred): the maintainer working from evals/<name>/prompt.md, evals/<name>/rubric.md, and the source files inside each eval directory. Exit criteria: every eval source path exists; with-shell prompts list source paths without duplicating large source content; no-shell prompts include enough preview content to complete the task.

  2. 2

    Normalize task rubrics

    Purpose: make success criteria explicit without turning prompts into hidden answer keys. Changes: review each rubric.md on its own merits; keep rubrics focused on task quality, source fidelity, and the Birch contract; decide whether rubrics are generation guidance, post-hoc scoring guidance, or both; if rubrics are shown to models, expose them consistently across evals. Owner / inputs (inferred): the maintainer using the per-eval rubric files and the benchmark harness behavior. Exit criteria: each eval has a clear rubric; harness behavior is documented, including whether rubric paths are for with-shell only or rubric content is included for no-shell when needed.

  3. 3

    Verify skill packaging

    Purpose: ensure the benchmark tests the published skill shape instead of an accidental local setup. Changes: keep checker and finish scripts inside skill/scripts/; keep template and CSS assets inside the skill package; ensure skill/SKILL.md references in-package paths; ensure snapshots include skill/, evals/, relevant scripts, docs, and styles. Owner / inputs (inferred): the skill package maintainer and the benchmark snapshot set. Exit criteria: a fresh model can read skill/SKILL.md, create an artifact, finish it, and run the checker without guessing paths; snapshot inputs are sufficient to reproduce a run.

  4. 4

    Run the final benchmark

    Purpose: produce comparable outputs for the selected model list. Changes: run the five evals across the chosen models; capture generation traces, deterministic checker reports, screenshots, and VLM review findings; record self-check attempts from actual tool calls, not from prompt mentions. Owner / inputs (inferred): the benchmark runner, the selected model list, and the packaged eval inputs. Exit criteria: each model has five artifacts or a clear generation failure; reports include deterministic findings, VLM findings, token and tool metrics, and the prompt files; the final analysis clearly separates rendering correctness from semantic artifact quality.

Risks

What can go wrong, and how to reduce it

Historical Birchline wording survives in prompts

Risk: models copy old classes or old artifact habits instead of the Birch recipe shape.

Mitigation: keep source briefs current and include an explicit unsupported-class list.

Prompts mix file contents with file paths

Risk: the prompt becomes self-contradictory and models cannot tell what must be read from disk versus what is already previewed.

Mitigation: use two prompt modes, one with shell paths only and one with no-shell source previews.

Models ignore the skill workflow

Risk: they hand-roll HTML instead of following the intended finish-and-check flow.

Mitigation: keep the normal fast-agent skill block, tell models to read SKILL.md, and validate the final artifacts.

Checker availability differs across runs

Risk: a benchmark pass cannot be reproduced because the checker is not packaged with the run.

Mitigation: keep the checker under skill/scripts/ and snapshot it with the run.

Deterministic checks miss semantic quality

Risk: the artifact can render correctly but still answer the task poorly.

Mitigation: add a later scoring pass that uses source, prompt, artifact, and task rubric.

Rubrics drift across evals

Risk: comparison becomes noisy because the scoring surface is not consistent.

Mitigation: review each rubric before the final benchmark and document how rubrics are exposed.

Verification

Before/after evals and what to compare

Before

  • Run the current birch-html skill against the five evals using the with-shell prompt mode.
  • Capture deterministic checker output, screenshots, VLM notes, token usage, duration, and self-check evidence.

After

  • Rerun the same five evals after the migration work lands, using the same harness and the same viewport checks.
  • Compare the outputs side by side to isolate what changed because of the recipe migration rather than because of the benchmark setup.

Checks to run in both passes

  1. 1

    Deterministic checks

    Run at desktop, mobile, deep, and mobile-deep viewports.

  2. 2

    Render evidence

    Capture screenshots and VLM review for visible layout defects.

  3. 3

    Comparison dimensions

    Compare generation success, deterministic failures and warnings, VLM findings, token usage, duration, and self-check evidence.

  4. 4

    Optional later scoring

    If needed, run a semantic scoring pass against each eval rubric after the generation and rendering pipeline is stable.

What the first benchmark pass must answer

  • Did the model produce complete standalone HTML for all five evals?
  • Did it follow the Birch skill workflow and finish the artifact correctly?
  • Did the artifact avoid unsupported Birchline classes?
  • Did deterministic and VLM checks find rendering problems?
  • Did the content remain source-grounded and useful for the target audience?
Non-goals

Work to defer for now

  • Do not port Birchline CSS wholesale.
  • Do not add compatibility aliases before eval evidence shows they are needed.
  • Do not build the future slot-based assembler before the skill benchmark is stable.
  • Do not treat VLM or rendering checks as full semantic content scoring.
