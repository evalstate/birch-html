Module explainer

How the rendering checker turns Birch HTML into findings, screenshots, and reports

This checker is deterministic-first: it parses the HTML contract, then optionally captures browser evidence, then writes JSON and Markdown so a contributor can change the checker without accidentally breaking 390px evals.

artifact mode pair mode optional Chrome reports + screenshots
Mental model

Three gates, one output contract

1. Static contract gate HTML + CSS shape

`page_stats()` and `contract_findings()` read the document before any browser work. They check the doctype, viewport meta, `.page` shell, CSS variable hygiene, Birch class coverage, and hard layout contracts like `.metric-row`, `.diff-row`, and `.flow-step`.

2. Visual smoke gate Chrome when available

If a browser is found, the checker captures screenshots, measures document height for deep viewports, and runs a geometry audit so 390px overflow and underfilled blocks show up as findings instead of surprises.

3. Report gate JSON + Markdown

The final payload records summary counts, per-artifact or pair findings, optional screenshot metrics, geometry notes, and pair deltas. That is what downstream evals and review scripts read back.

Caveat

If `--capture` is on but Chrome is missing, artifact mode adds a note and skips visual smoke checks; pair mode leaves screenshots and geometry empty. Keep this fallback in mind when you compare local and CI behavior.

Runtime path

From input pairs to reports and screenshots

Source-grounded path from scripts/check_birch_renderings.py: `main()`, `check_artifact()`, `compare_pair()`, `page_stats()`, `contract_findings()`, `capture()`, `screenshot_metrics()`, `geometry_audit()`, and `render_markdown()`. Inline chart helper code for consumer artifacts lives in scripts/birch_mpl.py.

  1. 1

    Choose mode and viewport

    `main()` accepts `--artifact` for generated eval outputs or `--pair` for reference:candidate comparisons. If you pass neither, `parse_pairs()` falls back to the default design-system and component-variants pairs. The viewport parser expects `name:widthxheight`, such as `desktop:1365x900`.

  2. 2

    Read page stats first

    `page_stats()` parses the HTML with `StatsParser`, counts tags, classes, data attributes, words, style blocks, and embedded Birch CSS, and compares used CSS variables against both the system stylesheet and local CSS. This is where the checker learns whether the candidate is structurally Birch-shaped.

  3. 3

    Run static contract findings

    `contract_findings()` is the main extension point for markup rules. It checks the doctype, viewport meta, `.page` shell, embedded or linked Birch CSS, zero unknown CSS vars, and several contract-specific parsers: `GridListParser`, `ChecklistParser`, `MetricRowParser`, `DiffRowParser`, `DiffPolarityParser`, `FlowStepParser`, and `TimelineParser`.

  4. 4

    Capture browser evidence when possible

    `find_chrome()` looks for `google-chrome`, `google-chrome-stable`, `chromium`, or `chromium-browser`. If one exists and capture is enabled, `capture()` writes PNGs into `reports/birch-screenshots/`. `capture_height_for_viewport()` measures document height for viewports whose name contains `deep`, so the screenshot can include below-the-fold content without wasting pixels on empty space.

  5. 5

    Convert pixels into findings

    Artifact mode uses `artifact_screenshot_metrics()` and `artifact_screenshot_findings()` to detect blank renders, palette drift, and blackish pixels. Pair mode uses `screenshot_metrics()` plus `screenshot_findings()` to compare source and candidate PNGs, then runs `geometry_audit()` and `geometry_findings()` for overflow, metric alignment, split-rail ordering, numeric header alignment, and wrapped code/diff problems.

  6. 6

    Write the report set

    `main()` serializes a JSON payload with `mode`, `summary`, and either `artifacts` or `pairs`, then writes the matching Markdown summary. Pair mode includes `original_stats`, `candidate_stats`, `deltas`, `screenshot`, `geometry`, and `findings`; artifact mode includes `stats`, `screenshot`, `geometry`, and `findings`.

File tour

Read the code in this order

  1. 1

    `scripts/check_birch_renderings.py`

    Start here. It owns the CLI, parsing, page stats, the contract checks, browser capture, screenshot metrics, geometry audit, and the final JSON/Markdown writers.

  2. 2

    `styles/birch-system.css`

    Read this next so the checker’s class expectations make sense. The source parser and the browser audit both assume the Birch primitives already define `.page`, `.section`, `.panel`, `.card`, `.metric-row`, `.flow-step`, `.numeric-table`, `.chart-panel`, `.checklist`, `.diff-row`, and `.timeline`.

  3. 3

    `scripts/birch_mpl.py`

    This is the chart helper layer. It applies the Birch Matplotlib theme and exports inline SVG without hiding Matplotlib, which matters for any artifact that uses generated charts.

  4. 4

    `evals/charts/build_chart_brief.py`

    This consumer shows the runtime path in miniature: generate an HTML artifact, inline SVG, numeric tables, and a report-ready page shell, then let the checker validate the result.

  5. 5

    `evals/charts/run_eval.py`

    This harness rebuilds the chart brief and reruns the checker at `desktop:1365x900`, `mobile:390x900`, `deep:1365x2400`, and `mobile-deep:390x3000`, writing per-viewport reports into `reports/`.

  6. 6

    `docs/birch-llm-style-guide.md`

    Use this when you need the authoring contract in words rather than code. It explains the Birch shell, token choices, list contracts, chart rules, and the mobile-safety assumptions that the checker later enforces.

Where to change behavior

Tune the checker in `scripts/check_birch_renderings.py` first: `DEFAULT_PAIRS`, `contract_findings()`, `compare_stats()`, `artifact_screenshot_findings()`, `screenshot_findings()`, `geometry_audit()`, `geometry_findings()`, `capture_height_for_viewport()`, and `render_markdown()` are the main knobs.

Extension points to remember

  • DEFAULT_PAIRS changes the default reference:candidate baseline.
  • BIRCH_CLASSES, LAYOUT_CLASSES, and SEMANTIC_CLASSES drive class-count warnings and notes.
  • contract_findings() is where new HTML rules, gradient bans, and CSS-variable checks belong.
  • artifact_screenshot_findings() and screenshot_findings() set the visual thresholds.
  • geometry_audit() and geometry_findings() control the viewport-level overflow and alignment checks.
Commands and outputs

What to run, and what it writes

uv run --with matplotlib python evals/charts/build_chart_brief.py
uv run --with pillow python scripts/check_birch_renderings.py \
  --artifact eval-runs/skill-with-shell-codexresponses-gpt-5-4-mini-publication-final/module-explainer.html \
  --out reports/module-explainer-check.json \
  --markdown reports/module-explainer-check.md \
  --viewport desktop:1365x900
uv run --with pillow python evals/charts/run_eval.py

Expected report files

  • reports/birch-rendering-check.json or the path you pass with --out; it stores mode, summary, and per-artifact or per-pair records with findings.
  • reports/birch-rendering-check.md or your --markdown path; it mirrors the JSON in a reviewer-friendly summary.
  • reports/birch-screenshots/<stem>-<viewport>.png; pair mode also emits *-diff.png and *-contact.png.
  • The chart harness writes reports/birch-rendering-check-chart-brief*.json and .md files for each viewport name in VIEWPORTS.

JSON payload shape

  • Artifact mode: {"mode":"artifact","artifacts":[...],"summary":{...}}.
  • Pair mode: {"mode":"pair","pairs":[...],"summary":{...}}.
  • Both modes count failures, warnings, and notes so downstream checks can decide whether to block or just surface smoke issues.
Gotchas

Things that can break an eval without changing the visuals much

Structural contracts are strict

  • .metric-row must be exactly caption + meter + code.
  • .diff-row must be span.ln + span.mark + span.code.
  • .flow-step must keep the number separate from one content wrapper.
  • .timeline-item needs timeline-time + timeline-body unless you scope a custom timeline class.

Color and token hygiene matters

  • Use Birch tokens and canonical palette values; unknown CSS variables fail static checks.
  • Page-local gradient backgrounds are rejected.
  • Literal hex colors are only a note unless they stray from the canonical palette enough to matter.

Browser evidence is optional, not imaginary

  • Chrome discovery happens through four binary names, in order.
  • Deep viewports can extend capture height, but the cap is still bounded by the requested viewport.
  • Pair mode adds screenshot diffs and contact sheets; artifact mode does not compare against a source image.
