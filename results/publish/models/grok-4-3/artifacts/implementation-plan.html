Implementation plan

Birchline → Birch recipe migration: recommended sequence

Grounded in evals/implementation-plan/migration-plan.md. First action: run Phase 1 to make the five benchmark evals reproducible before touching recipes or packaging.

What to do first

Phase 1 — Clean benchmark inputs

Purpose. Make every eval source path exist, prompts list repository paths without duplicating large content, and no-shell prompts contain enough preview material to finish the task without filesystem access.

Work

  • Keep each eval’s prompt.md, rubric.md, and source material together under evals/<name>/.
  • Remove historical or archived data sources from active eval paths.
  • Ensure prompts do not mention stale package names or deleted files.
  • Decide which prompts expose source previews and which require shell reads.
Exit criteria
  • Every eval source path exists.
  • With-shell prompts list source paths without duplicating large source content.
  • No-shell prompts include enough source preview content to complete the task.
Owner: maintainer running the benchmark harness.
Sequence

Milestones and phases

Phase 2Normalize task rubrics

Purpose. Make success criteria explicit without turning prompts into hidden answer keys.

  • Review each rubric.md on its own merits.
  • Keep rubrics focused on task quality, source fidelity, and Birch contract.
  • Document harness behaviour: rubric path only for with-shell; rubric content included for no-shell if needed.
Exit: each eval has a clear rubric; exposure rules documented.
Phase 3Verify skill packaging

Purpose. Ensure the benchmark tests the published birch-html skill shape.

  • Keep checker and finish scripts inside skill/scripts/.
  • Ensure skill/SKILL.md references in-package paths only.
  • Snapshot includes skill/, evals/, scripts, docs, and styles.
Exit: fresh model can read SKILL.md, create artifact, finish it, and run checker without guessing paths.
Phase 4Run final benchmark

Purpose. Produce comparable model outputs across the five evals.

  • Run numeric-data, code-review, module-explainer, implementation-plan, benchmark-comparison.
  • Capture generation traces, deterministic checker reports, screenshots, VLM findings.
  • Record self-check attempts from actual tool calls.
Exit: each model has five artifacts or clear failure; reports separate rendering correctness from semantic quality.
Grounding

Confirmed facts vs. assumptions & open questions

Confirmed facts (from migration-plan.md)

  • Package name is birch-html; skill entrypoint is skill/SKILL.md.
  • Preferred workflow: read sources → copy skill/resources/template.html → replace body → run finish_birch_html.py → validate with check_birch_renderings.py.
  • Artifact shell must be <main class="page stack" data-gap="lg">.
  • Final artifact is standalone HTML with embedded Birch CSS in style[data-birch-system].
  • Five benchmark evals listed; each has prompt.md + rubric.md under evals/<name>/.
  • Final benchmark must test the skill runner, not the older repo-mode path.
  • Keep .auto-grid + .card as the card-grid primitive; recipes remain structural, not decorative.
Assumptions
  • Final benchmark evaluates the birch-html skill workflow.
  • Shell-enabled runs may inspect local repository files.
  • No-shell runs remain possible with source previews.
  • Semantic output scoring can be added after the generation/rendering pipeline is stable.
Open questions
  • Should task rubrics be shown to models for every eval, or used only for human scoring?
  • Should the no-shell prompt set be part of the published benchmark or just a fallback/debug mode?
  • Which models should be included in the final published run?
  • Should semantic scoring be manual, model-judged, or hybrid?
Risk register

Risks and mitigations

1
Historical Birchline wording causes models to copy old classes
Mitigation: keep source briefs current and include explicit unsupported-class list in every prompt.
2
Prompt includes both file content and file paths, causing conflicts
Mitigation: maintain two prompt modes — with-shell paths only; no-shell source previews.
3
Models ignore skill workflow and hand-roll HTML
Mitigation: keep normal fast-agent skill block, instruct models to read SKILL.md, validate final artifacts.
4
Checker availability differs across runs
Mitigation: package checker under skill/scripts/ and snapshot it with the run.
5
Deterministic checks miss semantic quality
Mitigation: add later scoring pass using source, prompt, artifact, and task rubric.
6
Rubrics become inconsistent across evals
Mitigation: review each rubric before final benchmark and document exposure rules.
Verification

Verification plan (before/after evals)

  1. Generate artifacts with current birch-html skill and with-shell prompts for all five evals.
  2. Run deterministic checks at desktop, mobile, deep, and mobile-deep viewports.
  3. Capture screenshots and VLM review for visible layout defects.
  4. Compare generation success, deterministic failures/warnings, VLM findings, token usage, duration, and self-check evidence.
  5. Optionally run later semantic scoring pass against each eval rubric.

Key questions the first benchmark pass must answer

  • Did the model produce complete standalone HTML for all five evals?
  • Did it follow the Birch skill workflow and finish the artifact correctly?
  • Did the artifact avoid unsupported Birchline classes?
  • Did deterministic and VLM checks find rendering problems?
  • Did the content remain source-grounded and useful for the target audience?
Scope

Non-goals / deferred work

Do not port Birchline CSS wholesale.

Do not add compatibility aliases before eval evidence shows they are needed.

Do not build the future slot-based assembler before the skill benchmark is stable.

Do not treat VLM/rendering checks as full semantic content scoring.
