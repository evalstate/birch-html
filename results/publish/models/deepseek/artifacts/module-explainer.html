Module explainer

The rendering checker is a deterministic HTML/visual smoke pipeline that catches Birch contract violations before they ship.

A three-stage pipeline — static parsing, headless screenshot capture, and browser geometry audit — turns a single HTML artifact into a structured JSON/Markdown report. Source-grounded from scripts/check_birch_renderings.py, scripts/birch_mpl.py, evals/charts/, and the Birch CSS contracts in styles/birch-system.css.

Python Chrome headless Pillow Static HTML parsing
Mental model

Pipeline flow from main() in scripts/check_birch_renderings.py. In --artifact mode (preferred for generated eval outputs), the screenshot analysis and geometry audit run against a single file; in --pair mode, they compare reference and candidate.

What the checker actually does

At its core, check_birch_renderings.py is a deterministic, parse-first, browser-augmented validator. It does not use vision models, ML, or heuristics trained on prior artifacts. It reads the HTML file as text, parses it with specialized HTMLParser subclasses, then optionally launches Chrome to capture a screenshot and run a geometry audit script. Every finding is a structured (level, name, evidence) triple produced by a named check function.

The checker was designed to be simpler and more deterministic than the neighboring Birchline tooling. Its static checks are pure Python html.parser.HTMLParser subclasses — no DOM, no layout engine, no heuristic thresholds beyond explicit CSS minimums. The browser stage adds visual smoke detection that static parsing cannot see: blank renders, overflowing text, misaligned metric bars, and wrapping pathology.

Runtime path

From input paths to findings

  1. 1

    Parse CLI args and resolve inputs

    main() in scripts/check_birch_renderings.py resolves --artifact (single-file) or --pair (reference:candidate) paths relative to the repo root. In artifact mode, each path is checked independently; pair mode compares stats and screenshots between original and candidate. The two modes are mutually exclusive.

  2. 2

    Run static HTML/CSS contract checks

    page_stats() feeds the HTML through StatsParser to count tags, classes, CSS vars, words, and style bytes. Then contract_findings() runs a battery of checks: doctype, viewport meta, Birch CSS embedding (≥10 KB with all five signatures), .page shell, layout/semantic primitive counts, unknown CSS variables, patch marker lines, gradient prohibition, palette color purity, and SVG chart mark correctness.

  3. 3

    Run component-specific contract parsers

    Seven focused HTMLParser subclasses validate child-count contracts for Birch components: GridListParser (insight/takeaway list children), ChecklistParser (inline markup in checklist rows), MetricRowParser (caption + meter + code), DiffRowParser (ln + mark + code columns), DiffPolarityParser (add/del class or data-kind), FlowStepParser (flow-num + one wrapper), and TimelineParser (timeline-time + timeline-body).

  4. 4

    Launch headless Chrome for screenshot capture

    capture() opens the HTML file via file:// URI in Chrome --headless=new with --no-sandbox. For deep viewports, capture_height_for_viewport() measures the document height first via an injected script that sets data-birch-doc-height, then captures at the measured height (capped at the viewport max). Screenshots land in reports/birch-screenshots/.

  5. 5

    Analyze screenshots with Pillow

    In artifact mode, artifact_screenshot_metrics() sub-samples the PNG, computes background/non-background/blackish fractions, and checks for blank renders (file <2 KB or <2.5% non-background). In pair mode, screenshot_metrics() computes per-pixel RGB difference, generates an amplified diff PNG and a contact sheet.

  6. 6

    Run browser geometry audit

    geometry_audit() injects a ~500-line JS script that probes the live layout: overflow detection, container text spill, stat-card squeeze, metric bar alignment, timeline marker position, code-block wrapping pathology, and prose text underfill. Results are serialized into data-birch-audit on the body and extracted from Chrome's --dump-dom output.

  7. 7

    Emit JSON and Markdown reports

    All findings are collected into a payload with a summary hash (mode, artifact/pair count, failures, warnings, notes). render_markdown() produces a human-readable Markdown file with per-artifact tables; the raw JSON is written to the --out path. The script exits with code 1 on failures (or on warnings when --fail-on-warn).

File tour

Reading order for new contributors

Each file is described with its role in the checker pipeline, the key functions/classes to read first, and how it connects to the others. Start at the top.

1

scripts/check_birch_renderings.py

~1,400 lines · The main checker

This is the heart of the pipeline. Start with main() to see the top-level flow, then read check_artifact() (the artifact-mode entry point) and contract_findings() (the static check battery). The HTMLParser subclasses — StatsParser, GridListParser, ChecklistParser, MetricRowParser, DiffRowParser, DiffPolarityParser, FlowStepParser, TimelineParser — are each ~40–80 lines and validate a single component contract. The browser integration lives in find_chrome(), capture(), geometry_audit(), and geometry_findings().

Key functions: main, check_artifact, contract_findings, page_stats, geometry_audit, geometry_findings, svg_visual_findings.

scripts/birch_mpl.py

~120 lines · Matplotlib helpers

Provides the Birch Matplotlib theme used by artifact generators like build_chart_brief.py. Read rc_params() for the token-to-Matplotlib color mapping, birch_theme() (a context manager that applies rcParams), polish_axes() for grid/spine cleanup, and svg_string() for serializing a figure as inlineable SVG. evals/charts/build_chart_brief.py

Generates 24-birch-chart-brief.html from evals/charts/sample-data.csv. This is the best example of what a checked artifact looks like: it loads CSV rows, builds a Matplotlib SVG via birch_mpl, assembles KPI cards, metric rows, and a numeric table, then writes the complete Birch HTML. 4

evals/charts/run_eval.py

~50 lines · Eval runner

Ties generation and checking together. Runs build_chart_brief.py first, then calls check_birch_renderings.py --artifact at four viewports (desktop, mobile, deep, mobile-deep). Each viewport produces its own JSON and Markdown report. This is the pattern to follow when adding new eval artifacts.

Key: the VIEWPORTS dict and the run() loop.

5

docs/birch-llm-style-guide.md

~460 lines · The contract being checked

This is the authoritative LLM contract that the checker enforces. It defines required shell structure, layout primitives, semantic components, CSS variable tokens, and strict child-count contracts for diff rows, flow steps, metric rows, and timeline items. When you add a new check to the checker, it should correspond to a rule in this guide.

Key sections: Required page shell, quick primitive index, diff, timeline, flow steps, metric rows, charts and numeric data, rules for LLM generation.

6

styles/birch-system.css

~1,100 lines · The CSS that implements the contracts

The canonical stylesheet that the checker verifies is embedded (or linked). The checker asserts the embedded Birch CSS block is ≥10 KB and contains the five signature strings: --color-ivory, --font-serif, .page, .section, .card. When geometry audit probes measure overflow or wrapping, they are testing against the layout defined here.

Key: the CSS variable tokens section, the layout primitives responsive breakpoints (860px, 620px), and the component styles for diff, timeline, flow-step, metric-row.

Gotchas

Things that will surprise you

HTMLParser is not a DOM

Every parser subclass implements its own depth-tracking stack. There is no shared tree, no CSS cascade, and no computed style. The handle_starttag / handle_endtag / handle_data callbacks are streaming callbacks — you must track nesting state manually. This is why every parser has its own self.depth, self.in_* flag, and self.*_stack.

Chrome needs the real file path

The checker opens the HTML via file:// URI pointing at the actual repo path so that relative styles/birch-system.css links resolve. The geometry audit injects a <base href="..."> element into a temp copy to preserve these relative links. If you move the repo or rename directories, screenshots will capture unstyled HTML.

Diff polarity requires both the mark and the class

A + in .mark is visible but does not trigger green row styling. DiffPolarityParser checks that rows with + or - in the mark column also have .add / .del class or data-kind="add/del". The checker fails rows that rely on mark text alone.

SVG polyline defaults to black fill

Browsers fill un-styled <polyline> and <path> elements with black. svg_visual_findings() scans for chart-line elements that lack fill="none" or an explicit stroke attribute. This is a common LLM generation failure that produces large black wedges instead of line charts.

Metric alignment is measured in the browser

The geometry_audit() JS measures the left offset of every .meter within a .metric-list. A spread >2 px across meters in the same list produces a warning. This cannot be detected statically because it depends on the rendered width of label text.

Code wrap detection uses tree-walker rects

The geometry audit's rectStatsForText() walks text nodes with document.createTreeWalker and groups getClientRects() into visual lines. It compares visual line count to logical (newline-split) line count. If a code block that should wrap produces 3× more visual lines than logical lines with <18 chars per line, it is flagged as underfilled.

Extension points

Where to add or change behavior

Add a new static contract check

Find the relevant class in BIRCH_CLASSES, LAYOUT_CLASSES, or SEMANTIC_CLASSES in check_birch_renderings.py (lines ~60–100) and add it if missing. Then add a new check(...) call inside contract_findings() (line ~400). If the check requires custom HTML parsing, write a new HTMLParser subclass (~40–80 lines) following the pattern of MetricRowParser or FlowStepParser:

  • Track entry/exit with a boolean flag and a depth counter.
  • Collect direct children (depth=1 elements).
  • Validate the child contract in handle_endtag when depth returns to 0.
  • Append to a self.bad_* list with a preview snippet.

Wire the new parser into contract_findings() and emit a Finding when the bad-items list is non-empty.

Add a new geometry audit probe

Inside the audit_js string in geometry_audit(), add a new querySelectorAll loop that measures element positions, sizes, or overflow. Serialize results into the data-birch-audit JSON object. Then add a corresponding block in geometry_findings() that reads the key from the parsed JSON and emits Finding objects. The pattern is: if problems: emit fail; elif key exists: emit pass; else: skip (pre-audit-checker baseline).

Add a new supported viewport

Add an entry to the VIEWPORTS dict in evals/charts/run_eval.py (or your own eval runner). The format is "name": "label:WxH". The checker's parse_viewport() splits on : and x. For "deep" viewports (name contains "deep"), the checker measures document height before capturing; for standard viewports it uses the exact height.

Change palette/token validation

The canonical palette lives in PALETTE (line ~100 of check_birch_renderings.py) and is duplicated in COLORS in birch_mpl.py. Update both if you change the Birch palette. The is_palette_hex() function uses a tolerance of 3 per channel. palette_metrics() uses tolerance 34 for the "palette-close" fraction. Adjust these tolerances in the checker only; the CSS token values in birch-system.css are the source of truth.

Change the Birch CSS contract signature

The five strings in BIRCH_CSS_SIGNATURES (line ~40) are what the checker scans for to validate the embedded Birch CSS block. If you add or rename a critical CSS variable or class, update this tuple. The BIRCH_CSS_MIN_BYTES threshold (10,000) is also defined there. These signatures are matched against the raw CSS text, not parsed selectors.

Key constants

BIRCH_CSS_MIN_BYTES 10,000

Minimum embedded Birch CSS block size. Defined at line ~40 of the checker.

SEMANTIC_CLASS_MIN 6

Minimum distinct semantic component classes. Below this, the artifact likely uses page-specific CSS.

LAYOUT_CLASS_MIN 8

Minimum layout primitive uses. Below this, the layout may be brittle.

PALETTE_TOLERANCE 3/ch

Per-channel RGB tolerance for is_palette_hex(). Palette-metrics tolerance is 34.

Finding levels

  • fail — hard contract violation; blocks adoption
  • warn — likely problem, non-blocking unless --fail-on-warn
  • note — informational delta or diagnostic
  • pass — check passed (emitted only for geometry/audit checks)

Source and caveat

All claims are grounded in the six files listed in evals/module-explainer/source-files.txt, inspected at the time of writing. The checker is a living tool; function line numbers may drift. The geometry audit JS is the most complex and fragile part — it probes live browser layout and depends on the Birch CSS being loaded correctly. If Chrome or the Birch CSS changes, audit probes may need adjustment. The --pair mode is maintained for backward compatibility but --artifact mode is preferred for generated eval outputs.
