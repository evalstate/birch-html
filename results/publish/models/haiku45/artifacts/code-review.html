Candle rendering checker: structural validation enhancements

Patch adds three new HTML parser validators and enhanced geometry audits to the Candle rendering checker. Findings-first review identifies three blocking severity issues and four medium/low-risk observations.

3 blocking issues Scope: validation logic, severity regression Files: scripts/check_candle_renderings.py + 2 new files
Blocking findings

Gate-keepers: must resolve before merge

FAIL Severity regression: layout_overflow check

scripts/check_candle_renderings.py, line ~847

Evidence 
- "warn",        # line 828 (original)
+ "fail",        # line 847 (new)
  "layout_overflow",
  "; ".join(...for item in worst)
Impact

Breaking change. Layouts with minor overflow (>2px, e.g., rounding artifacts in text wrapping or badge badge edges) now fail hard instead of warn. Existing compliant designs could gate-keep unexpectedly.

Root cause

The 2-pixel threshold detects sub-pixel rendering, layout shifts, and typography variations without distinguishing severity. Changing to fail treats all overflow equally, but desktop layouts often have 1-3px overflow from kerning, rounding, or browser rendering differences that don't affect readability.

Recommended fix (choose one)
  • A (safest): Revert to warn for layout_overflow.
  • B (recommended): Increase threshold from 2px to 4px or 5px to filter cosmetic overflow.
  • C (alternative): Keep fail level but add opt-out via data-candle-no-overflow-check for known cosmetic overflow.
FAIL Overly strict data visualization requirement

scripts/check_candle_renderings.py, lines ~613–623

Evidence 
numeric_rich = (
    stats.class_counts.get("numeric-table", 0) > 0
    and (stats.class_counts.get("stat-value", 0) >= 3
         or stats.class_counts.get("metric", 0) >= 12)
)
chart_count = (
    stats.class_counts.get("chart-panel", 0)
    + stats.class_counts.get("chart-svg", 0)
    + stats.tag_counts.get("svg", 0)
)
if numeric_rich and chart_count == 0:
    findings.append(
        Finding("fail", "missing_data_visualization", ...)
    )
Impact

Fails reports with 3+ KPI cards and 12+ metric cells but no charts. A valid numeric-only brief (KPI strip + table) would fail even if it correctly communicates data. Forces unnecessary chart generation for table-first report designs.

Problem

Rule assumes numeric-rich always needs visualization. But table-only briefs are a valid communication pattern (e.g., numeric comparison tables, ranked metrics without trends). This check treats them as incomplete.

Recommended fix (choose one)
  • A (safest): Change "fail" to "note" or remove entirely. Tables are sufficient.
  • B (recommended): Downgrade to "warn" and only flag if >20 metrics with no chart.
  • C: Add allowlist via class attribute (e.g., data-no-chart-required) for intentional table-only reports.
FAIL TimelineParser custom timeline class detection incomplete

scripts/check_candle_renderings.py, lines ~230–243

Evidence 
if int(item["depth"]) == 1:
    children = item["children"]
    assert isinstance(children, list)
    children.append(classes)
# ... later ...
timeline_classes = item["timeline"]
is_custom = len(timeline_classes - {"timeline"}) > 0
if not ((has_time and has_body) or is_custom):
    self.bad_items.append(...)
Problem

Checks for custom classes on the outer .timeline container, but not on individual .timeline-item elements. A timeline-item with custom scoped classes (e.g., <div class="timeline-item custom-scope">) will fail validation even if it's intentionally using local CSS instead of time/body structure.

Impact

False failures on valid custom timeline implementations. Maintainers with item-level scoping patterns must work around the validator. Valid HTML that deviates from the expected structure for good reason gets gate-kept.

Recommended fix
  • Extend is_custom check: also allow items that have custom classes at the item level (not just timeline level). Change line ~240 to: is_custom = len(timeline_classes - {"timeline"}) > 0 or len(item["children"][0] - {"timeline-item", "tl-entry"}) > 0
  • Or: store item-level classes separately and check them in validation logic.

WARN JavaScript geometry audit uses unchecked positioning assumption

scripts/check_candle_renderings.py, lines ~824–842 (JavaScript audit_js)

Evidence 
var dotLeft = parseFloat(dotStyle.left);
var itemRect = item.getBoundingClientRect();
var dotCenter = itemRect.left + dotLeft + 5.5;
var expected = timelineRect.left + 9.75;
var delta = Math.round((dotCenter - expected) * 10) / 10;
if (Math.abs(delta) > 2.5) {
    // fail
}
Assumption

Assumes timeline-item uses position: relative and ::before is positioned relative to item's left edge. If item uses position: absolute, position: sticky, or pseudo-element is positioned differently, calculation breaks.

Impact

False negatives: real misaligned timeline markers won't be detected if positioning context differs. The hardcoded 5.5px offset will fail silently on uncommon but valid timeline patterns.

Recommended fix

Before calculating delta, check that item's computed position is relative. If not, skip this check and log a note instead of failing silently.

WARN MetricRowParser does not validate meter element internal structure

scripts/check_candle_renderings.py, lines ~189–211

Current validation

Checks that .metric-row has exactly 3 direct children: caption + .meter + code. But doesn't validate that .meter contains exactly one <span> with a style attribute.

Example that passes but shouldn't 
<div class="metric-row">
  <span class="caption">Pass rate</span>
  <div class="meter">
    <span style="--value: 98%"></span>
    <span style="--value: 98%"></span>  <!-- extra span -->
  </div>
  <code>0.98</code>
</div>
Impact

Invalid meter structures could silently fail CSS rendering later. Maintainers would see visual issues but not know the root is malformed HTML.

Recommended fix

Extend MetricRowParser to inspect meter's children: must contain exactly one element (span, div, etc.) with a style attribute. Log as "note" if meter has 0 or 2+ children.

Reviewer checklist

Gate criteria and verification steps

Before merge

  • Blocking issue #1: Severity regression (layout_overflow) has been addressed. Confirm threshold change or revert decision is documented.
  • Blocking issue #2: Data visualization check downgraded to warn or made optional. Verify table-only briefs won't fail.
  • Blocking issue #3: TimelineParser item-level custom class detection has been extended or threshold raised.
  • Medium-risk #4: JavaScript timeline positioning check includes validation that item uses position: relative.
  • Medium-risk #5: MetricRowParser validates meter internal structure or is documented as known limitation.

Recommended follow-up tests

  • Run patch against existing chart-brief examples to confirm layout_overflow severity change doesn't gate-keep valid designs.
  • Test MetricRowParser against metric-rows with extra spans or missing style attributes.
  • Test TimelineParser with custom-scoped timeline-items (e.g., .timeline-item.custom-scope without time/body).
  • Verify CSV fixture file exists and error message is helpful if missing.
  • Test geometry audit on timeline with position: absolute or position: sticky items.
Residual risks

Known issues if merged as-is

  • Gate-keeping: Layouts with 2–3px overflow (common on desktop due to typography rounding) will fail hard. Existing compliant designs may need workarounds.
  • False failures: Table-only numeric briefs will fail the visualization check. Valid reports require chart addition even if not needed for communication.
  • Custom timelines: Item-level scoping will be flagged as invalid. Maintainers with custom timeline patterns must refactor or disable check.
  • Silent geometry failures: Timelines with uncommon positioning will not be audited; marker misalignment goes undetected.
Summary

Merge recommendation

Status: Request changes

The patch adds valuable structural validation to the Candle rendering checker, but introduces three blocking severity issues that require resolution before merge:

  1. Severity regression on layout_overflow (warn → fail) with a 2px threshold that will gate-keep valid designs.
  2. Overly strict data visualization rule that fails table-only numeric briefs.
  3. TimelineParser custom class detection incomplete, leading to false failures on valid item-level scoping.

Recommend resolving all three blocking issues and addressing the medium-risk geometric audit assumptions before landing. The parser architecture and code quality are solid; the issues are policy/threshold decisions, not structural problems.
