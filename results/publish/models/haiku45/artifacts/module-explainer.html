Module explainer

Rendering Checker Pipeline

A deterministic static and visual smoke test for Birch HTML artifacts, designed to catch contract violations, layout overflow, and palette drift before human review.

Contract validation Screenshot metrics Geometry audit
Mental model

How the checker works

The rendering checker takes one or more HTML artifacts and validates them through three stages:

  1. 1

    Static contract parsing

    Read HTML, extract CSS, count tags and classes, verify Birch system CSS is embedded or linked. Check for known problematic patterns: loose text in grid items, missing polarity markers on diffs, malformed timeline structures.

  2. 2

    Screenshot capture (optional)

    If Chrome is available, render the HTML at a specified viewport (desktop, mobile, or deep page) and compare visual distance using Pillow image metrics. Measure palette coherence and background saturation.

  3. 3

    Geometry audit (optional)

    Inject a large JavaScript block into the page before capture. The script measures text overflow, metric alignment, timeline markers, and flow-step geometry. Reports layout problems that static parsing cannot detect.

All findings (pass/fail/warn/note) are collected into a JSON report and rendered as Markdown for human review. The tool exits with code 1 if there are failures or warnings (via --fail-on-warn).

Code tour

Reading order for new contributors

  1. 1

    scripts/check_birch_renderings.py — entry point and main logic

    Read from line 1 to understand the module docstring. Then jump to main() to see argument parsing, mode selection (artifact vs. pair), and report generation. Follow the flow: it calls either check_artifact() or compare_pair() for each input.

  2. 2

    Data classes: Finding and PageStats

    Around lines 50–100, see the @dataclass definitions. Finding has level, name, evidence. PageStats collects all static facts about a page: tag counts, class counts, CSS variables, byte sizes, etc.

  3. 3

    Parser classes: StatsParser, GridListParser, etc.

    Lines 140–450+. Each parser is a custom HTMLParser subclass that walks the DOM looking for specific patterns. StatsParser counts everything; others find misuse of grid-based lists, checklists, metric rows, diffs, flow steps, and timelines. Each parser has a bad_items or bad_rows list that becomes a Finding if non-empty.

  4. 4

    contract_findings() — the contract checker

    Around line 650. This function calls all the parsers and returns a list of Findings. It checks: doctype, viewport, Birch CSS presence, page shell, primitive count, unknown CSS vars, patch markers, gradient gradients, and runs all the specialized parsers. Add new checks here if you need to validate new patterns.

  5. 5

    page_stats() — static HTML analysis

    Around line 600. Reads HTML, uses StatsParser to extract facts, extracts CSS blocks, checks for Birch CSS validity (size, presence of signatures). Returns a PageStats dataclass that becomes part of the JSON report.

  6. 6

    check_artifact() and compare_pair()

    Around lines 550–600. Entry points for the two modes. check_artifact() calls page_stats() and contract_findings(), then optionally captures and audits. compare_pair() does the same for both files, then calls compare_stats() for delta analysis and screenshot_metrics() to measure visual distance.

  7. 7

    Screenshot and geometry code

    capture() around line 1100 uses Chrome. screenshot_metrics() uses Pillow. geometry_audit() around line 1450 is the largest function; it injects a massive JavaScript audit block and extracts the results. geometry_findings() converts audit data to Findings.

  8. 8

    render_markdown() — report generation

    Around line 1800. Takes the JSON payload and generates a Markdown report. Useful for understanding the structure of the final report and adding new sections if you extend findings.

  9. 9

    Reference: docs/birch-llm-style-guide.md and styles/birch-system.css

    These define what's valid in a Birch artifact. The style guide lists all Birch classes and patterns; the CSS is the source of truth for what's available. When adding contract checks, consult these files to ensure you're validating against the canonical rules.

Gotchas

Common traps and edge cases

Loose text in grid-based lists

A .insight-list item must wrap content in exactly one child. Direct text becomes an extra grid item and wraps one-word-per-line. The GridListParser detects this by counting direct children depth.

Fix: Always wrap prose in <span> or <div>.

Diff-row polarity markers

A .diff-row.add must have either a .add class OR a data-kind="add" attribute. A visible + in the .mark span alone is not enough; Birch colors rely on the class or data attribute. DiffPolarityParser checks for this mismatch.

Fix: Use both the visual marker and the semantic attribute.

Timeline structures

A .timeline-item must contain either .timeline-time + .timeline-body or use a custom scoped timeline class. The default two-column grid breaks if the structure is wrong. TimelineParser validates this before rendering.

Fix: Always provide time + body children unless you override the grid in page-local CSS.

Flow-step layout

A .flow-step is a two-column grid: .flow-num on the left, content on the right. If content is loose (not wrapped in a <div>), it becomes an extra grid child and can wrap one-character-per-line in narrow viewports. FlowStepParser catches this.

Fix: Always wrap content after .flow-num in exactly one container.

SVG chart presentation

A <polyline> or chart line without fill="none" renders as a solid black wedge (SVG default). The checker looks for explicit fill: none in CSS or inline fill="none" attribute, and also checks for a stroke. Missing either is flagged as svg_chart_mark_presentation fail.

Fix: Always set fill="none" and define a stroke color on chart line marks.

Chrome not found

Screenshot capture requires Chrome (or Chromium). The checker calls find_chrome() which searches google-chrome, google-chrome-stable, chromium, chromium-browser. If none exist, screenshot and geometry audits are skipped, but the checker continues (only a note-level finding).

Fix: Install Chrome/Chromium, or use --capture false if you only care about static checks.

Pillow not installed

If PIL/Pillow is not available, screenshot_metrics() returns None and all Pillow-based checks are skipped. The JSON report will show null for metrics but the run continues.

Fix: pip install Pillow or use a UV script block with --with pillow.

Extension points

Where to change behavior

Add a new contract check

Add a parser class (inherit from HTMLParser) and wire it into contract_findings(). Example:

class MyCustomParser(HTMLParser):
    def __init__(self): ...
    def handle_starttag(self, tag, attrs): ...
    # returns self.bad_items

parser = MyCustomParser()
parser.feed(html)
if parser.bad_items:
    findings.append(Finding("fail", "my_check", ...))

Change the Birch CSS signatures

Edit the BIRCH_CSS_SIGNATURES tuple if the system CSS no longer includes certain class names or variables. The checker uses these to validate that the stylesheet was embedded correctly.

Current: ("--color-ivory", "--font-serif", ".page", ".section", ".card")

Modify the palette for color checks

The PALETTE dict (around line 90) lists canonical colors as RGB tuples. is_palette_hex() uses this to check if a literal hex color is close enough to a palette color. Update if the palette changes or if you want stricter/looser color matching.

Add or remove screenshot viewports

In evals/charts/run_eval.py, the VIEWPORTS dict defines which viewport tests to run. Add new sizes (e.g., "tablet: 768x1024") and the runner will capture and check them. The --viewport argument to the main script accepts name:WxH format.

Customize geometry audit checks

The geometry_audit() function contains a 1000+ line JavaScript block that measures layout. To add new checks, edit the audit_js variable, add a new measurement, append it to the data-birch-audit JSON, then add a corresponding handler in geometry_findings().

Change report output format

The JSON schema is defined by how we build the payload dict in main(). The render_markdown() function reads this payload and generates Markdown. Edit render_markdown() to change the Markdown output format, or add a new render function for a different output type (e.g., HTML, YAML).

Adjust screenshot comparison thresholds

artifact_screenshot_findings() and screenshot_findings() check for hardcoded thresholds like if background_fraction < 0.025 or if palette_close < 0.40. Edit these functions to loosen or tighten the thresholds.

Example commands

Concrete usage and outputs

Run a single artifact check

python3 scripts/check_birch_renderings.py \
  --artifact 24-birch-chart-brief.html \
  --out reports/check-chart.json \
  --capture true \
  --viewport desktop:1365x900

Run pair comparison (reference vs candidate)

python3 scripts/check_birch_renderings.py \
  --pair 05-design-system.html:05-design-system-birch.html \
  --fail-on-warn

Run without screenshots (static checks only)

python3 scripts/check_birch_renderings.py \
  --artifact 24-birch-chart-brief.html \
  --capture false

Run the chart eval fixture

cd /path/to/birch-html
uv run --with matplotlib python evals/charts/run_eval.py

This builds the chart HTML, then runs the checker at four viewports (desktop, mobile, deep variants).

  • styles/birch-system.css — referenced in all artifacts; checked by the validator
  • docs/birch-llm-style-guide.md — source of truth for valid classes and patterns

  • evals/charts/run_eval.py — orchestrates build + check
  • evals/charts/build_chart_brief.py — generates test fixture HTML

The checker is designed to be self-contained. It can run standalone on any HTML artifact without needing the generator or other eval infrastructure.

Summary

The rendering checker is a deterministic, contract-first validator for Birch HTML artifacts. It combines static HTML parsing, optional screenshot capture, and in-browser geometry audits to catch layout and styling issues before human review. New contributors should start by reading the main entry point, understanding the parser pattern, and familiarizing themselves with Birch primitives in the style guide.

For production use, integrate the checker into your eval pipeline via evals/charts/run_eval.py or similar orchestrator. Always run with --fail-on-warn in CI to catch regressions early.
