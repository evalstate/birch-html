Implementation plan · eval: implementation-plan

Birchline → Birch recipe migration sequencing for benchmark readiness

Grounded in evals/implementation-plan/migration-plan.md, this plan is a practical execution order for the maintainer: what to do first, what is confirmed, what is assumed, how to gate each phase, and how to verify quality before final run.

Grounded source framing

First action and decision posture

Recommendation

Start with Phase 1: Clean benchmark inputs. The file explicitly places this phase first to reduce migration noise, make eval paths reproducible, and make rubrics and packaging checks deterministic.

Source-grounded status

Confirmed vs assumed vs unresolved

Confirmed facts
  • Package is named birch-html and workflow is driven by skill/SKILL.md.
  • Canonical workflow currently: read sources → copy template → replace body → run uv run skill/scripts/finish_birch_html.py → validate with check_birch_renderings.py when available.
  • Published artifact shell is fixed: <main class="page stack" data-gap="lg">.
  • Final artifact must remain standalone with inline Birch CSS in style[data-birch-system].
  • Target evals are exactly five: numeric-data, code-review, module-explainer, implementation-plan, benchmark-comparison.
  • Each eval has prompt.md and rubric.md; source data is inside eval directories.
  • Migration decision: keep .auto-grid + .card (no .card-grid alias yet).
  • Keep skill-based assembly as current next-step; slot-based assembler remains deferred.
  • Migration goals are structural/semantic, not decorative: recipes should preserve behavior without Birchline visual systems.

Execution plan

Milestones / phases with inputs, purpose, and exit criteria

  1. 1

    Phase 1 — Clean benchmark inputs

    Purpose: Make each eval path reproducible and reduce stale context before semantic work.

    • Changes: keep each eval prompt/rubric/source together; remove historical archived sources from active eval paths; align prompts with current names/paths; define source-preview policy by prompt mode.
    • Owner / inputs: maintainer; evals/, prompts, rubrics, source files.
    • Exit criteria: every eval source path exists; shell prompts list file paths only where expected; no-shell prompts provide enough preview context.
  2. 2

    Phase 2 — Normalize task rubrics

    Purpose: Keep success conditions explicit without exposing hidden answer keys.

    • Changes: review each rubric.md, enforce quality/source-fidelity/contract focus, classify rubric visibility for with-shell vs no-shell usage.
    • Owner / inputs: maintainer + prompt designers; all evals/*/rubric.md.
    • Exit criteria: each eval has clear rubric; harness behavior around rubric visibility is documented.
  3. 3

    Phase 3 — Verify skill packaging

    Purpose: Ensure benchmark uses the published skill contract, not a legacy path.

    • Changes: keep checker/finish scripts under skill/scripts; keep template/CSS in package; ensure skill/SKILL.md points to in-package paths; include enough snapshot inputs.
    • Owner / inputs: maintainer + skill authors; skill/ tree and snapshot config.
    • Exit criteria: a fresh model can read SKILL.md, generate artifact, run finish, run checker without path guessing.
  4. 4

    Phase 4 — Run final benchmark

    Purpose: generate comparable outputs and collect deterministic + visual evaluation signals.

    • Changes: run five evals across selected models, capture traces, checker reports, screenshots, VLM findings, and prompt files.
    • Owner / inputs: benchmark operator + selected model set (decision point); eval artifacts + scripts.
    • Exit criteria: each model has five outputs or explicit failure notes; reports include deterministic + VLM findings + token/tool metrics; analysis separates rendering from semantic quality.
Risks

Risks and mitigations

Risk

Birchline-style wording regresses model style

Legacy terms or classes can reappear even after migration if prompts are stale.

Mitigation: keep migration prompts current and include unsupported-class list in every mode.

Risk

Prompt shape conflicts with shell modes

Combining file content and file paths in one mode can reduce signal quality.

Mitigation: enforce two explicit modes (with-shell paths only; no-shell with source previews).

Risk

Workflow drift outside SKILL

Generated outputs may bypass the skill flow and fail deterministic checks.

Mitigation: keep skill invocation required, run checker after generation, and reject outputs that miss shell workflow artifacts.

Risk

Checker noise vs semantic quality

Deterministic checks may miss semantic correctness or over-flag non-semantic layout changes.

Mitigation: add semantic scoring pass after rendering pipeline stabilizes (per source plan).

Verification

Before/after eval verification plan

Before state (current baseline)

What to capture

  • Generate artifacts with current birch-html skill and with-shell prompts.
  • Track all five benchmark members explicitly: numeric-data, code-review, module-explainer, implementation-plan, benchmark-comparison.
  • Run deterministic checks in desktop, mobile, deep, and mobile-deep viewports.
  • Capture screenshots and VLM layout review findings.
  • Collect generation success, deterministic warnings/failures, token usage, duration, and self-check artifacts.
After state (post-migration)

What should improve

  • All five evals are fully generated, each with complete standalone HTML.
  • All outputs follow Birch shell + local stylesheet + no unsupported classes.
  • Deterministic and VLM findings are stable or improved on the same checkpoints.
  • Source-groundedness remains verifiable through paths/evidence included in each artifact.
Pass/Fail gates
  • Did model produce standalone HTML for all five evals?
  • Did it follow skill workflow and finish artifact correctly?
  • Did output avoid Birchline-only classes?
  • Did checker + VLM find layout defects?
  • Did content stay source-grounded and useful?

Source: migration-plan.md verification section.

Optional quality extension

If stable after rendering pass, run a later semantic scoring pass against each eval rubric, as suggested in the source plan.

Scope control

Non-goals / deferred work

  • Do not port Birchline CSS wholesale.
  • Do not add Birchline compatibility aliases until evaluator evidence requires them.
  • Do not build the future slot-based assembler before benchmark stability is proven.
  • Do not treat VLM/rendering checks as full semantic content scoring.
Bottom line

Go decision for maintainer

Gate the sequence through Phase 1 → Phase 4; only then run the final benchmark. This keeps migration scoped, keeps evidence comparable, and avoids adding speculative infrastructure before the benchmark validates the current skill workflow.
