Module explainer

Rendering checker pipeline

Source-grounded contributor guide for the checker path that validates Birch rendering artifacts, captures screenshots, and emits JSON/Markdown findings.

deterministic-first browser-backed geometry safe output contract
Mental model

One pipeline, two execution modes

Core idea

The checker works in two phases: first deterministic HTML/CSS contract checks, then optional browser evidence (screenshots + in-page geometry audit). In candidate-only mode, it does not compare against a reference image; it validates that the artifact is styled, structured, and rendered without obvious breakage.

Mode is set by arguments, not by content: --artifact or --pair.

Input abstraction

Mode branch is early and strict

--artifact checks generated outputs only; --pair compares reference and candidate.

If both are provided, main() exits with an argument error.

Static stage

Parser-based contract checks

page_stats() and parsers collect counts, classes, variables, and style metrics. contract_findings() enforces shell, viewport, Birch primitives, and component-contract rules.

This includes parsers for .metric-row, .diff-row, .flow-step, .timeline.

Evidence stage

Optional screenshot + geometry evidence

capture() writes PNGs. geometry_audit() injects a script and returns page-wide overflow, wrap, rail-alignment, and timeline-metric geometry signals.

If no Chrome is available, visual checks are skipped with notes instead of fails.

Report stage

Findings and hard exits

The same payload is emitted to both JSON and Markdown. Non-zero exit happens on fails, or with --fail-on-warn when warnings are present.

Schema is intentionally stable: mode, summary, item list + findings list.

Runtime path

From input pairs to reports, screenshots, and findings

  1. 1

    Argument parse and mode branch

    main() reads --artifact, --pair, --viewport, --capture, --delay-ms, and --fail-on-warn. Mixed modes are rejected.

  2. 2

    Target set construction

    If --pair is empty, defaults are used: 05-design-system.html:05-design-system-birch.html and 06-component-variants.html:06-component-variants-birch.html. Artifact mode uses the exact paths passed by --artifact.

  3. 3

    Static extraction

    page_stats() parses HTML with StatsParser, counts tags/classes/words, detects local vars, and checks Birch-system CSS usage with BIRCH_STYLE_RE patterns.

    contract_findings() applies required checks like doctype, viewport meta, .page shell, Birch class density, and component contracts.

  4. 4

    Pair mode extras

    In pair mode, compare_pair() adds deltas and structure/text drift checks before visual capture. In artifact mode, check_artifact() emits only the single artifact row plus optional visual checks.

    Screenshot capture and geometry run on candidate/stem path(s) after contract checks pass.

  5. 5

    Browser evidence path

    If Chrome is found, capture() uses --headless=new --screenshot. File naming is consistent by stem and viewport suffix, for example 24-birch-chart-brief-desktop.png.

    For deep viewports, height may be adapted via capture_height_for_viewport() using a measured document height script.

  6. 6

    Payload assembly and output

    All findings and metrics are serialized to --out (JSON) and --markdown (human report). Summary counters are computed from all item findings.

This flow avoids brittle image-only judging in the first stage, then adds visual checks only where HTML structure cannot detect raster breakage.

File tour

Read order for safe edits

1) scripts/check_birch_renderings.py

Start here: entrypoint, CLI parsing, mode selection, and all static/visual checks.

  • Must read: main(), check_artifact(), compare_pair().
  • Then: page_stats() and contract_findings().
  • Then: compare_stats(), deltas(), geometry_audit().
  • Last: render_markdown() and failure/exit policy.

2) evals/charts/run_eval.py

Execution driver for the chart-eval path.

  • Calls uv run --with matplotlib python evals/charts/build_chart_brief.py.
  • Then runs the checker four times in artifact mode with different viewport names.
  • Good first passpoint when extending viewport and report naming conventions.

3) evals/charts/build_chart_brief.py

Generates fixture HTML and embeds chart SVG.

  • Loads evals/charts/sample-data.csv.
  • Builds Matplotlib chart via birch_mpl.birch_theme().
  • Writes 24-birch-chart-brief.html.

4) scripts/birch_mpl.py

Rendering policy shared by chart generation.

  • rc_params() and birch_theme() set global style.
  • polish_axes() and svg_string() standardize chart output.

5) styles/birch-system.css

Defines layout primitives and semantic contracts used by the checker.

  • Tokens: var(--bg), var(--surface), var(--text).
  • Primitives: .page, .stack, .grid, .split, .card, .panel.
  • Flow utilities: .flow-list, .flow-step, .flow-node, .flow-edge.

6) docs/birch-llm-style-guide.md

Source of component expectations and safe command/code-list patterns.

  • Read contract guidance before changing generated output shape.
  • Use .chart-panel + .numeric-table for exact data, not inline chart hacks.

Gotchas and extension points

What to change, and where to avoid regressions

High-risk gotchas

  • Do not weaken defaults in BIRCH_CSS_MIN_BYTES or BIRCH_CSS_SIGNATURES without re-running existing fixtures.
  • Do not change parser contracts (GridListParser, FlowStepParser, TimelineParser) without updating expectations in source artifacts.
  • Do not remove --pair default behavior unless external orchestration owns those references.
  • Do not alter output key names in payload lightly; downstream consumers can be key-based.
  • Do not bypass --viewport handling for deep pages; capture_height_for_viewport() is there to prevent truncation or oversized captures.

Safe extension points

  • Add checks: add to contract_findings() and keep finding names stable.
  • Threshold tuning: update artifact_screenshot_findings(), screenshot_findings(), and geometry_findings().
  • Diff logic: update compare_pair() when introducing new pair metadata fields.
  • Viewport coverage: modify VIEWPORTS in evals/charts/run_eval.py and output naming conventions simultaneously.
  • Style policy: update scripts/birch_mpl.py when chart rendering expectations need tighter consistency.
Contributor checklist
  • Keep all edits grounded in tokens and primitives: var(--accent), .page, .stack, .flow-step, .numeric-table-wrap.
  • When adding long commands or findings examples, use code-block with data-wrap="true" so mobile stays readable.
  • Run: python3 scripts/check_birch_renderings.py --artifact 24-birch-chart-brief.html --out ... after any checker-touching change.
  • Prefer updating and validating both report channels: JSON + markdown.
