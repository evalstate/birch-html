Module explainer

Rendering checker pipeline

How scripts/check_birch_renderings.py validates Birch artifacts — from HTML input to findings, screenshots, and reports.

Static contract checks Pillow screenshot metrics Browser geometry audit --artifact and --pair modes
Start here

Mental model

The checker is a deterministic, layered pipeline. Each layer catches a different class of failure:

  1. 1

    Static HTML/CSS contract

    HTMLParser-based checks for doctype, viewport meta, embedded Birch CSS (≥10,000 bytes with required signatures), .page shell, layout primitive density, semantic component density, undefined CSS variables, forbidden patch-marker lines (+ at line start), gradient backgrounds, and canonical palette colors.

  2. 2

    Component contract validation

    Eight specialized HTMLParser subclasses validate grid-child contracts: .insight-list/.takeaway-list items must have one direct child (else they become accidental grid columns); .checklist items must not have multiple direct children when the CSS sets display: flex; .metric-row must be caption + .meter + code; .diff-row must be .ln + .mark + .code; +/- diff rows need .add/.del class or data-kind attribute; .flow-step must have .flow-num + one content wrapper; .timeline-item needs .timeline-time + .timeline-body; chart SVG polyline/path must have fill=none and explicit stroke.

  3. 3

    Screenshot capture & Pillow metrics

    Chrome headless captures a PNG. Pillow measures palette-close fraction (how much of the image uses canonical Birch palette colors), ivory fraction, and blackish fraction. In --pair mode it computes mean RGB delta and RMS delta between original and candidate.

  4. 4

    Browser geometry audit

    An in-page JS audit runs after load. It measures overflow (text spilling outside boxes), container text overflow, stat card squeeze, section rail ordering, timeline marker alignment, metric bar x-position alignment across a .metric-list, metric row value clipping, split rail heading alignment, numeric table header alignment, wrapped code/diff underfill (lines consuming <45% of container width), and pathological text wrapping (<14 chars/visual line).

  5. 5

    Findings → JSON + Markdown reports

    All findings (fail/warn/note) are collected and written to reports/birch-rendering-check.json and reports/birch-rendering-check.md. Screenshots go in reports/birch-screenshots/. Contact sheets and amplified diffs are generated in --pair mode.

Key design decision

The checker is intentionally simpler than a full visual regression suite. Screenshots are evidence-capture and smoke tests, not pixel-perfect diffs. Most checks are deterministic static analysis that runs without a browser.

Runtime path

From input pairs to reports

Artifact mode (generated evals)

python scripts/check_birch_renderings.py \
  --artifact 24-birch-chart-brief.html \
  --out reports/birch-rendering-check-{name}.json \
  --viewport desktop:1365x900

Single candidate artifact. Captures screenshot, runs contract + geometry checks, writes one report entry. No diff against a reference.

Pair mode (development / before-after)

python scripts/check_birch_renderings.py \
  --pair 05-design-system.html:05-design-system-birch.html \
  --pair 06-component-variants.html:06-component-variants-birch.html \
  --capture \
  --viewport desktop:1365x900

Original vs candidate. Captures both screenshots, computes RGB delta, generates contact sheet and amplified diff PNG. Default pairs are defined in the script's DEFAULT_PAIRS tuple.

Multi-viewport eval runner

python evals/charts/run_eval.py

Builds the chart brief first, then runs the checker at four viewports: desktop:1365x900, mobile:390x900, deep:1365x2400, mobile-deep:390x3000. Outputs four JSON/MD report pairs.

Output artifacts

  • reports/birch-rendering-check.json — structured findings payload
  • reports/birch-rendering-check.md — human-readable summary table
  • reports/birch-screenshots/{stem}-{viewport}.png — captured renderings
  • reports/{orig}__vs__{cand}-diff.png — amplified diff (pair mode)
  • reports/{orig}__vs__{cand}-contact.png — source/candidate/diff contact sheet (pair mode)

JSON payload shape

{
  "mode": "artifact",          // or "pair"
  "artifacts": [...],          // only in artifact mode
  "pairs": [...],              // only in pair mode
  "summary": {
    "artifacts": 1,            // checked artifacts (artifact mode)
    "pairs": 1,                // checked artifacts (artifact mode) OR pairs (pair mode)
    "failures": 0,
    "warnings": 2,
    "notes": 3
  }
}

Each artifact/pair entry has findings[], stats (PageStats), optional screenshot metrics, and optional geometry audit results.

Pipeline visualization

Checker flow

  1. ①

    Parse CLI args

    Resolve --artifact (preferred for eval) or --pair. Read styles/birch-system.css to build the set of defined CSS variables.

  2. ②

    Page stats extraction

    page_stats() runs StatsParser on each HTML file. Collects tag counts, class counts, CSS vars used/defined, hex colors, embedded Birch CSS byte count and validity.

  3. ③

    Contract findings

    contract_findings() runs eight specialized parsers and structural checks. Each failure emits a Finding(level, name, evidence). Undefined CSS vars, missing shell, too few primitives, bad grid children, bad diff polarity, missing chart SVG, SVG fill/stroke defaults — all caught here.

  4. ④

    Screenshot capture (optional)

    If Chrome is available and --capture is set, capture() runs Chrome headless with --screenshot. Document height is measured first via a temp HTML injection to avoid blank deep captures.

  5. ⑤

    Pillow screenshot metrics

    In artifact mode: artifact_screenshot_metrics() → palette-close fraction, ivory fraction, blackish fraction. In pair mode: screenshot_metrics() → mean RGB delta, RMS delta, amplified diff PNG, contact sheet.

  6. ⑥

    Browser geometry audit

    geometry_audit() injects a JS audit script, runs Chrome headless with --dump-dom, parses data-birch-audit JSON from the output. Measures overflow, container overflow, timeline marker alignment, metric bar spread, stat card squeeze, split rail heading alignment, code underfill, pathological text wrapping.

  7. ⑦

    Findings aggregation

    geometry_findings() converts audit JSON into Finding objects. artifact_screenshot_findings() or screenshot_findings() converts Pillow metrics into findings. All findings merged into the report payload.

  8. ⑧

    Write reports

    render_markdown() generates a table of findings per artifact/pair. JSON and Markdown written to disk. main() exits with code 1 if any failures (or warnings when --fail-on-warn is set).

Source tour

File reading order for new contributors

Read these files in order. Each one builds context for the next.

1. styles/birch-system.css

Start here

Read this first to understand the design system. The checker validates that artifacts use classes and CSS variables defined in this file. Key sections: CSS custom properties (design tokens), layout primitives (.page, .stack, .grid, .split, etc.), semantic components (.card, .metric-row, .diff-row, .timeline-item, etc.), chart and numeric components. The checker's BIRCH_CSS_SIGNATURES, BIRCH_CLASSES, LAYOUT_CLASSES, SEMANTIC_CLASSES, and PALETTE constants are derived from this file.

Also read: docs/birch-llm-style-guide.md for the contract that LLMs must follow when generating artifacts.

2. scripts/check_birch_renderings.py — main() and entry point

Core

Read the main() function first. It shows the overall flow: parse args → find Chrome → iterate artifacts or pairs → call check_artifact() or compare_pair() → aggregate findings → write reports. Then read compare_pair() and check_artifact() to understand the two modes. The Finding and PageStats dataclasses define the data model for all findings.

Key constants: DEFAULT_PAIRS, BIRCH_CSS_SIGNATURES, BIRCH_CSS_MIN_BYTES=10_000, SEMANTIC_CLASS_MIN=6.

3. scripts/check_birch_renderings.py — page_stats() and StatsParser

Static analysis

StatsParser is the workhorse HTML parser. It collects: page title, tag counts (tag_counts), class counts (class_counts), CSS variable definitions and usage, hex colors, embedded Birch CSS byte count, doctype presence, viewport meta presence. page_stats() runs it and returns a PageStats dataclass. All other parsers are variants of this pattern.

4. scripts/check_birch_renderings.py — component contract parsers

Land mines

Read all eight contract parsers in order. Each targets a specific grid/flex layout that breaks when content is not wrapped correctly:

  • GridListParser — .insight-list/.takeaway-list items with multiple direct children collapse to 1–2 characters per line in narrow columns because they become extra grid children
  • ChecklistParser — .checklist rows with multiple direct children wrap incorrectly when the CSS sets display: flex
  • MetricRowParser — .metric-row requires exactly three children: caption + .meter + code; any extra children break meter alignment
  • DiffRowParser — .diff-row is a three-column grid; loose text auto-places into the narrow line-number column
  • DiffPolarityParser — +/- diff rows without .add/.del class or data-kind attribute render without green/red Birch coloring
  • FlowStepParser — .flow-step is a two-column grid; loose title/detail children auto-place into the narrow number column
  • TimelineParser — .timeline-item must use .timeline-time + .timeline-body (or a custom scoped class)

Also read svg_visual_findings() for the SVG chart fill/stroke default check.

5. scripts/check_birch_renderings.py — contract_findings()

Entry point for checks

This is the main function that orchestrates all static checks. It calls each parser, collects bad items, and emits Finding objects. To add a new static check, add a parser or a direct check here and append findings. The function signature is: contract_findings(candidate: Path, stats: PageStats, system_defined_vars: set[str]) → list[Finding].

6. scripts/check_birch_renderings.py — screenshot and geometry

Visual

Read capture(), capture_height_for_viewport(), measure_document_height(), screenshot_metrics(), artifact_screenshot_metrics(), palette_metrics(), and make_contact_sheet(). These handle Chrome-based screenshot capture and Pillow-based image analysis. Then read geometry_audit() — the large JS audit string is inline in the Python file. It runs in the browser and dumps a JSON object via data-birch-audit attribute.

The geometry audit is the most complex piece. It handles overflow detection, timeline marker alignment, metric bar spread, stat card squeeze, split rail heading alignment, code wrap underfill, and pathological text wrapping.

7. scripts/birch_mpl.py

Chart generation

Matplotlib helpers for Birch-styled inline SVG charts. Four public functions: rc_params() (returns Birch rcParams dict), birch_theme() (context manager), polish_axes() (removes top/right spines, adds y-axis grid), svg_string() (exports figure as inline SVG string), close() (closes Matplotlib figure). Palette: clay #D97757, olive #788C5D, sky #6A8CAF, rust #B04A3F.

Used by evals/charts/build_chart_brief.py to generate the chart fixture.

8. evals/charts/run_eval.py

Eval runner

The chart eval driver. Builds the chart brief first (using build_chart_brief.py), then runs the checker at four viewports. Each viewport produces a separate JSON and MD report. The VIEWPORTS dict shows all tested configurations.

9. evals/charts/build_chart_brief.py

Fixture generator

Reads evals/charts/sample-data.csv, generates a 24-birch-chart-brief.html fixture using scripts/birch_mpl.py. Uses .numeric-table, .metric-list, .chart-panel, .chart-svg with role="img" and aria-label. Demonstrates the complete Birch numeric artifact contract.

Land mines

Gotchas and where behavior hides

Grid list item children

.insight-list and .takeaway-list are CSS grids. Each <li> is a grid child. If an item has multiple direct children (e.g., <strong> plus loose text), the text becomes a second grid column. In a narrow rail, that column is ~60px and text wraps as one or two characters per line. The fix: wrap all item content in one <span> or <div>.

✓ <li><span><strong>Title.</strong> Detail text.</span></li>
✗ <li><strong>Title.</strong> Detail text.</li>

Checklist inline layout collision

.checklist li gets display: block !important from birch-system.css. But if the LLM adds .checklist li { display: flex } in page-local CSS, items with multiple direct children will wrap incorrectly. The checker detects this: it runs ChecklistParser and if bad items exist AND .checklist li { display: flex } is in the CSS, it emits a checklist_inline_layout fail.

Diff row child order

.diff-row is a three-column grid: grid-template-columns: 48px 18px minmax(max-content, 1fr). Loose text between <span> elements gets auto-placed into the narrow .ln column, producing a 1-character-per-line underfilled block. The checker validates the exact child sequence: tag/class of children[0] must include ln, children[1] must include mark, children[2] must be code. Exactly three children required.

Diff polarity metadata required

A + or - in .mark alone is not enough to get green/red row styling. The checker requires .add or .del class OR data-kind="add"/data-kind="del" attribute on the .diff-row. DiffPolarityParser checks this. Without it, added/deleted rows render as context (gray) rows.

Flow step content wrapper

.flow-step is a two-column grid: grid-template-columns: 42px minmax(0, 1fr). The first child must be .flow-num. All content must be in one wrapper element (e.g., <div>) in column 2. If <h3 class="flow-title"> and <p class="flow-detail"> are direct children of .flow-step, both get grid-placed: the title goes to column 1 (narrow), the detail goes to column 2, producing one-character-per-line in the title.

✓ <li class="flow-step">
    <span class="flow-num">1</span>
    <div>
      <h3 class="flow-title">...</h3>
      <p class="flow-detail">...</p>
    </div>
  </li>

Split rail section chrome alignment

CSS Grid aligns boxes, not semantic intent. If the first child of a .split starts with an .eyebrow, the side rail will align to that eyebrow instead of the content heading. The checker emits section_rail_order warning when the aside/reference rail appears before main content cards in the DOM. Fix: put section chrome above the .split so the rail aligns with the content heading, not the eyebrow.

Geometry audit needs <base> tag

geometry_audit() injects a <base href="{ROOT.resolve().as_uri()}/"> tag before the audit JS. This is critical: without it, relative stylesheet links resolve against the temp file's path and the audit runs against unstyled HTML. The code inserts the base before the audit JS, before </head>.

Document height measurement

Chrome's --screenshot captures the requested window height, not a stitched full page. capture_height_for_viewport() uses a two-pass approach: measure document height first via a temp HTML injection with --dump-dom and a data-birch-doc-height attribute, then capture at that height (capped). Without this, deep captures of short pages waste VLM budget on blank space, and tall pages get cropped.

SVG chart fill=none trap

LLMs often write <polyline points="..."> for charts without setting fill="none". Browsers default to fill: black, producing a large black wedge instead of a line chart. The checker runs svg_visual_findings() to catch chartish polyline/path elements lacking explicit fill="none" and explicit stroke. The CSS is also checked for .fill-none classes.

Gradient backgrounds forbidden

Birch artifacts use flat token surfaces. The checker detects linear-gradient, radial-gradient, or conic-gradient in local CSS (not in data-birch-system embedded blocks) and emits a no_birch_gradients fail. This is enforced in contract_findings() via a regex search on the extracted local CSS.

Metric row contract and narrow panels

In narrow reference panels, .metric-row content can overflow. The geometry audit measures this: if the value <code> element's text width exceeds its bounding rect, or if the row overflows the owner container, it emits metric_row_squeeze. Fix: keep labels and values short, or set compact --metric-label / --metric-value on the .metric-list.

Pair vs artifact mode output shape

The JSON payload has different structure in each mode. In --artifact mode: payload["artifacts"] holds the report entries and payload["summary"]["pairs"] = checked artifact count. In --pair mode: payload["pairs"] holds the report entries and payload["summary"]["pairs"] = pair count. External summary scripts use summary.pairs as the item denominator. Do not confuse the two modes.

Extensibility

Where to change behavior

Add a new static check

Subclass HTMLParser like the existing eight parsers, or add a direct regex/CSS check in contract_findings(). Append Finding objects to the list it returns. Set warn=True on check() for non-blocking warnings.

def contract_findings(...):
    # existing checks...
    my_parser = MyNewParser()
    my_parser.feed(html)
    if my_parser.bad_items:
        findings.append(Finding("fail", "my_new_check",
            "examples: " + " | ".join(my_parser.bad_items[:4])))
    return findings

Add a new geometry audit check

Extend the inline JS audit string in geometry_audit(). Add a new key to the returned JSON object. Then add a corresponding branch in geometry_findings() to convert the JSON into Finding objects.

# In geometry_audit JS string:
var newCheck = [];
document.querySelectorAll('.my-component').forEach(function(el) {
  // measure, push to newCheck
});
// Append to audit output object
document.body.setAttribute('data-birch-audit', JSON.stringify({
  // existing keys...
  newCheck: newCheck
}));

# In geometry_findings():
new_check = audit.get("newCheck") or []
if isinstance(new_check, list) and new_check:
    findings.append(Finding("fail", "new_check_name",
        "; ".join(f"{item['selector']} ..." for item in new_check[:5])))

Add a new viewport configuration

Add an entry to the VIEWPORTS dict in evals/charts/run_eval.py. Use the format "name:widthxheight". Special names: deep triggers document-height measurement for the capture height; mobile tests narrow layout.

VIEWPORTS = {
    "chart-brief": "desktop:1365x900",
    "chart-brief-mobile": "mobile:390x900",
    "chart-brief-deep": "deep:1365x2400",   # add new viewport here
    "chart-brief-mobile-deep": "mobile-deep:390x3000",
}

Change the Birch CSS signature requirements

The minimum byte count and required signature strings are in check_birch_renderings.py:

BIRCH_CSS_MIN_BYTES = 10_000
BIRCH_CSS_SIGNATURES = (
    "--color-ivory", "--font-serif",
    ".page", ".section", ".card"
)

These are used in page_stats() to set birch_css_valid and in contract_findings() for the uses_birch_system_css check.

Adjust palette tolerance for screenshot metrics

palette_metrics() uses a tolerance of 34 for palette-close detection and 18 for ivory detection. is_palette_hex() in static checks uses tolerance 3. These are tuned for the Birch palette; changing them affects palette_close_fraction thresholds in screenshot findings.

Add a new Birch class to the watch list

Update the constant sets in check_birch_renderings.py. BIRCH_CLASSES drives the birch_class_delta note in pair comparisons. LAYOUT_CLASSES and SEMANTIC_CLASSES drive density checks. SEMANTIC_CLASS_MIN=6 sets the minimum semantic component count.

Customize screenshot output paths

Screenshot directory is set via --screenshots-dir (default: reports/birch-screenshots). Filenames are derived from the HTML file stem plus viewport name. Diff and contact sheet filenames are derived from the original and candidate stems. All paths go through display_path() to show relative paths in JSON output.

uv run python scripts/check_birch_renderings.py \
  --artifact evals/output/24-birch-chart-brief.html \
  --out reports/birch-rendering-check.json \
  --markdown reports/birch-rendering-check.md \

uv run python scripts/check_birch_renderings.py \
  --pair 05-design-system.html:05-design-system-birch.html \
  --pair 06-component-variants.html:06-component-variants-birch.html \

Run full chart eval at all viewports

uv run python evals/charts/run_eval.py

Builds the chart brief fixture first, then runs four viewport checks. Output: reports/birch-rendering-check-{name}.json and .md for each viewport.

Fail on warnings too

uv run python scripts/check_birch_renderings.py \
  --artifact 24-birch-chart-brief.html \
  --fail-on-warn

Exit code 1 if any warnings are present, not just failures. Useful in strict CI pipelines.

Check available Chrome

which google-chrome || which chromium || echo "no chrome"

The checker looks for google-chrome, google-chrome-stable, chromium, or chromium-browser. If none is found, screenshot capture and geometry audit are skipped with a note.
