Birchline → Birch Recipe Migration

Sequencing the remaining work for the birch-html skill benchmark, separating facts from assumptions, and defining verification criteria.

Audience: The maintainer deciding how to sequence migration work. Assumes familiarity with the birch-html package structure and skill workflow.

What to do first

Start with Phase 1: Clean Benchmark Inputs

Before running any benchmark, the source material must be coherent. Phase 1 is the foundation—everything else depends on reliable eval paths and clean prompts. Skipping it means running benchmarks against stale or ambiguous inputs.

  • Remove archived/stale data sources from active eval paths
  • Audit prompts for deleted files or outdated package references
  • Decide which prompts expose source previews (no-shell) vs. require shell reads

Migration phases

Phase 1 Clean Benchmark Inputs

Purpose: Make the benchmark easy to understand and reproducible.

Changes

  • Keep each eval's prompt, rubric, and source material together under evals/
  • Remove historical or archived data sources from active eval paths
  • Ensure prompts do not mention stale package names or deleted files
  • Decide which prompts expose source previews and which require shell reads

Inputs

Current eval directories, existing prompt files

Owner

Maintainer audit

Exit criteria

  • Every eval source path exists
  • With-shell prompts list source paths without duplicating large content
  • No-shell prompts include enough source preview to complete the task
Phase 2 Normalize Task Rubrics

Purpose: Make success criteria explicit without turning prompts into hidden answer keys.

Changes

  • Review each rubric.md on its own merits
  • Keep rubrics focused on task quality, source fidelity, and Birch contract
  • Decide whether rubrics are generation guidance, post-hoc scoring, or both
  • Expose rubrics consistently if shown to models

Exit criteria

  • Each eval has a clear rubric
  • Harness behaviour is documented: rubric path for with-shell, content for no-shell if needed
Phase 3 Verify Skill Packaging

Purpose: Ensure the benchmark tests the published skill shape.

Changes

  • Checker and finish scripts live inside skill/scripts/
  • Template and CSS assets live inside the skill package
  • skill/SKILL.md references in-package paths
  • Snapshots include skill/, evals/, relevant scripts, docs, and styles

Exit criteria

  • A fresh model can read SKILL.md, create an artifact, finish it, and run the checker without guessing paths
  • Snapshot inputs are sufficient to reproduce a run
Phase 4 Run Final Benchmark

Purpose: Produce comparable model outputs.

Changes

  • Run the five evals across the selected model list
  • Capture generation traces, deterministic checker reports, screenshots, and VLM review findings
  • Record self-check attempts from actual tool calls, not from prompt mentions

Exit criteria

  • Each model has five artifacts or a clear generation failure
  • Reports include deterministic findings, VLM findings, token/tool metrics, and prompt files
  • Final analysis clearly separates rendering correctness from semantic artifact quality

Risks and mitigations

Risk Mitigation
Historical Birchline wording causes models to copy old classes Keep source briefs current and include an explicit unsupported-class list
Prompt includes both file content and file paths, causing conflicts Use two prompt modes: with-shell paths only; no-shell source previews
Models ignore skill workflow and hand-roll HTML Keep normal fast-agent skill block, instruct models to read SKILL.md, and validate final artifacts
Checker availability differs across runs Package checker under skill/scripts/ and snapshot it with the run
Deterministic checks miss semantic quality Add a later scoring pass using source, prompt, artifact, and task rubric
Rubrics become inconsistent across evals Review each rubric before final benchmark and document how rubrics are exposed

Verification plan

Before / After Eval Comparisons

    Generate artifacts

  2. 2
    Run deterministic checks
    Test at desktop, mobile, deep, and mobile-deep viewports
  3. 3

    Screenshots and VLM review for visible layout defects
  4. 4
    Compare metrics
    Generation success, failures/warnings, VLM findings, token usage, duration, self-check evidence
  5. 5
    Optional scoring pass
    Evaluate against each eval rubric for semantic quality

First benchmark pass must answer

  • Did the model produce complete standalone HTML for all five evals?
  • Did it follow the Birch skill workflow and finish the artifact correctly?
  • Did the artifact avoid unsupported Birchline classes?
  • Did deterministic and VLM checks find rendering problems?
  • Did the content remain source-grounded and useful for the target audience?

Assumptions

  • The final benchmark is intended to evaluate the birch-html skill workflow
  • Shell-enabled runs are allowed to inspect local repository files
  • No-shell runs are optional but should remain possible with source previews
  • Semantic output scoring can be added after the generation/rendering pipeline is stable

Open questions

Rubric exposure: Should task rubrics be shown to models for every eval, or used only for human scoring?
No-shell scope: Should the no-shell prompt set be part of the published benchmark or just a fallback/debug mode?
Model selection: Which models should be included in the final published run?
Scoring method: Should semantic scoring be manual, model-judged, or hybrid?

Non-goals

  • Do not port Birchline CSS wholesale
  • Do not add compatibility aliases before eval evidence shows they are needed
  • Do not build the future slot-based assembler before the skill benchmark is stable
  • Do not treat VLM/rendering checks as full semantic content scoring
