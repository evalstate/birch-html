Module explainer

Rendering checker pipeline

A deterministic-first contract checker for Birch HTML artifacts. It parses static structure, validates component contracts, optionally screenshots via Chrome, measures pixel drift, and emits a JSON + Markdown report of pass/fail/warn/note findings.

Mental model

How the checker thinks

The checker is an HTML parser + optional browser audit that answers one question: does this artifact conform to the Birch contract and look right? It never executes JavaScript or runs a real DOM — it uses Python's HTMLParser subclass to walk tags and enforce structural invariants. If Chrome is available it adds a geometry audit that runs an in-browser JS snippet and harvests overflow metrics; if Pillow is available it also quantifies palette drift and blank renders from screenshots.

There are two modes. Artifact mode (--artifact) validates a standalone candidate; it screenshots, measures background/palette fractions, and flags blank or wildly off-palette renders without comparing to a reference. Pair mode (--pair) compares an original to a Birch-system candidate, computing RGB deltas and producing a contact sheet.

Runtime path

From input to report

Each phase appends Finding objects; the final payload is written to reports/birch-rendering-check.json and .md.

  1. 1

    CLI entry point

    main() parses --artifact or --pair flags, reads styles/birch-system.css for defined CSS variables, and resolves viewport spec (--viewport desktop:1365x900). Output paths default under reports/.

  2. 2

    Static parse + contract checks

    page_stats() runs StatsParser (a HTMLParser subclass) to count tags, classes, data-attributes, words, and style blocks. contract_findings() then validates: doctype, viewport meta, Birch CSS presence/signature, .page shell, layout primitive density, semantic component count, CSS variable validity, no gradient backgrounds, no raw patch markers, SVG fill/stroke on chart polypaths, and six component-contract parsers (GridListParser, ChecklistParser, MetricRowParser, DiffRowParser, DiffPolarityParser, FlowStepParser, TimelineParser).

  3. 3

    Chrome screenshot + pixel metrics

    If --capture is on and Chrome is found, capture() runs headless Chrome (--screenshot) against the artifact URI. In pair mode, screenshot_metrics() computes mean and RMS RGB deltas plus a 4× amplified diff image; in artifact mode, artifact_screenshot_metrics() measures background fraction, palette-close fraction, and blackish fraction. Both call palette_metrics() against the 12-entry PALETTE dict.

  4. 4

    Geometry audit

    geometry_audit() injects a long JS snippet into the artifact, opens it in headless Chrome with --dump-dom, and reads the data-birch-audit attribute from the DOM. The JS measures overflow, container-text overflow, stat-card squeeze, section-rail ordering, timeline geometry, metric alignment, metric-row squeeze, split-rail heading alignment, numeric-table header alignment, code-wrap underfill, and pathological text wrapping.

  5. 5

    Report output

    All Finding(level, name, evidence) objects are collected into a JSON payload with mode, per-artifact stats, screenshot metrics, geometry audit, and a summary dict counting failures/warnings/notes. render_markdown() writes a human-readable table. Exit code 1 if any failures (or warnings with --fail-on-warn).

File tour

Read order for new contributors

  1. 1

    styles/birch-system.css

    Start here. This is the single source of truth for all Birch primitives, tokens, and component styling. The checker validates that artifacts embed or link this file and that their local CSS variables are a subset of defined vars. Knowing what .page, .stack, .card, .flow-step, .metric-row, and .diff-row actually do will make every contract check obvious.

  2. 2

    docs/birch-llm-style-guide.md

    The LLM contract document. It defines the required shell, allowed primitives, component markup patterns, and generation rules. Many checker rules — "no gradients", ".metric-row must be caption + meter + code", ".diff-row must be ln + mark + code", insight-list items need exactly one child — originate here. Read it alongside the CSS.

  3. 3

    scripts/check_birch_renderings.py

    The main checker. Read main() first to see the CLI → mode selection → loop flow, then check_artifact() (artifact mode) and compare_pair() (pair mode). Then walk the parser classes: StatsParser, MetricRowParser, DiffRowParser, DiffPolarityParser, FlowStepParser, GridListParser, ChecklistParser, TimelineParser. Each is a small HTMLParser subclass that catches one class of structural bug. Finish with geometry_audit() which is the most complex function — it builds a temporary HTML file with injected JS, loads it in Chrome, and parses JSON from the DOM.

  4. 4

    scripts/birch_mpl.py

    Matplotlib theme helper for Birch-styled SVG charts. Not part of the checker itself, but artifacts the checker validates often include inline SVG generated by this module. Understanding birch_theme(), polish_axes(), svg_string(), and the COLORS dict helps you understand what valid chart SVG looks like and why the checker expects fill="none" on chart polylines.

  5. 5

    evals/charts/build_chart_brief.py

    A concrete artifact builder that uses birch_mpl.py to generate a chart-brief HTML page. It reads evals/charts/sample-data.csv, builds a Matplotlib line chart, inlines the SVG, and writes 24-birch-chart-brief.html. Read this to see how a well-formed Birch numeric artifact is assembled — it's a working example of every contract the checker enforces.

  6. 6

    evals/charts/run_eval.py

    The eval runner that wires everything together: it calls build_chart_brief.py to generate the HTML, then invokes check_birch_renderings.py --artifact at four viewports (desktop, desktop-deep, mobile, mobile-deep). This is the integration test for the full pipeline. Read it last.

Gotchas

Things that trip up contributors

  • Artifact mode and pair mode are mutually exclusive. Passing both --artifact and --pair raises SystemExit. The main() function checks this explicitly.
  • Chrome must be on PATH for screenshot + geometry checks. find_chrome() searches for google-chrome, chromium, etc. If absent, all visual checks emit a note-level finding and are skipped — the run still succeeds with only static checks.
  • Pillow is optional. If from PIL import Image fails at runtime, screenshot metrics silently return None and a note finding is emitted. Install with --with pillow when running via uv.
  • The geometry audit needs a real file URI. geometry_audit() injects <base href="…"/> so relative stylesheet links resolve. The artifact is written to a temp file; without a valid base href, the audit runs against unstyled HTML and produces false overflow findings.
  • BIRCH_CSS_MIN_BYTES is 10,000. The checker rejects embedded CSS shorter than 10 KB even if the placeholder is present. This catches incomplete postprocessing where the placeholder wasn't replaced. The five BIRCH_CSS_SIGNATURES (--color-ivory, --font-serif, .page, .section, .card) must also be present.
  • Local CSS over 2,500 bytes gets a warning. The threshold is in contract_findings(). Recurring patterns should move to birch-system.css, not stay in page-local style blocks.
  • Component-contract parsers don't validate semantics. MetricRowParser checks that .metric-row has exactly 3 direct children matching caption + meter + code — it does not verify the meter value or label text. Same shallow-structure approach for DiffRowParser, FlowStepParser, etc.
  • Pair mode defaults are hardcoded. DEFAULT_PAIRS lists 05-design-system.html and 06-component-variants.html pairs. If those files don't exist locally, the first check will be a fail with "missing" evidence.
Extension points

Where to change behavior

Add a new component-contract check

Create a new HTMLParser subclass alongside MetricRowParser etc. in check_birch_renderings.py. Call .feed(html) inside contract_findings(). Append a Finding on violation. Keep the parser stateless except for its bad_items list.

Add a new geometry audit check

Add a new JS query block inside the audit_js string in geometry_audit(). Push the results array onto the data-birch-audit JSON object. Then add a corresponding handler in geometry_findings() to turn the audit data into Finding objects.

Change the palette or color thresholds

Edit the PALETTE dict at module scope to add/rename palette entries. is_palette_hex() uses a tolerance of 3 RGB units (configurable via the tolerance parameter). palette_metrics() uses distance 34 for "close" and artifact_screenshot_metrics() uses 26 for "background". Change these thresholds in the respective functions.

Change report format or output paths

Override via CLI: --out and --markdown control report paths; --screenshots-dir controls image output. To change the JSON schema itself, edit the payload dict in main() and the rendering in render_markdown().

Add a new viewport or capture mode

New viewports go in the --viewport flag format name:WxH. Deep-capture logic (auto-extending height) lives in capture_height_for_viewport(), which checks if "deep" is in the viewport name. To add a new auto-height strategy, extend that function.

Caveat

This explainer is based on the source files listed in evals/module-explainer/source-files.txt as of the current commit. The checker's behavior is determined by its runtime environment (Chrome availability, Pillow installation, viewport flags) — not all checks run every time. When in doubt, read the source directly.
